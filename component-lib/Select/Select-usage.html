Select is the canonical single- or multi-select control for picking from a list of options too long to display inline. In 8x8 product surfaces it's the default for >5 options, for searchable or async lists, and any time the user needs a chip-style multi-select inside a form. Built on react-select — most react-select props pass through.

Best Practices

✅ When to use

  • There are more than five options to choose from (verbatim from docs).
  • You want to create a list of options for the user to choose (verbatim from docs).
  • The full list won't fit inline as Radio / Checkbox buttons.
  • Options need to be searched, filtered, or fetched from a server.
  • The user might pick many items at once — chip-style multi-select (isMulti).

❌ When not to use

  • Fewer than five options, single-select → use Radio (verbatim from docs)
  • Fewer than five options, multi-select → use Checkbox (verbatim from docs)
  • Binary toggle that takes effect immediately → use ToggleButton
  • Free-form input with suggestions → use an Autocomplete pattern (or isCreatable only if the closed list is the source of truth)
Placement (verbatim from the docs page): "Usually a select is used in a form element. By default, the width of the select is set to fixed. Alternatively, the width of the select is fluid as its parent container size."

Do / Don't

Pair 1 — Reach for Select when the list is too long for Radio / Checkbox
✅ Do — Use Select for >5 options or a searchable list

The docs page draws an explicit line: more than five options → Select. Below that line, Radio (single) or Checkbox (multi) wins because users compare alternatives side-by-side without an extra click.

Source: oxygen.8x8.com — Best Practices / When to use (verbatim) · examples.md line 167.

❌ Don't — Use Select for a 2 or 3 option choice

A Select hides options behind an extra click and replaces a one-glance comparison with an open / scan / close cycle. The docs page is explicit: "Otherwise, use a radio button for a single selection or a checkbox for multiple selections."

See Radio-usage.md Pair 4 for the inverse rule.

Pair 2 — Use isMulti (with hasCheckbox when helpful) for genuine multi-select
✅ Do — Render a single multi-select Select for one decision with several values

isMulti produces a chip-style display and emits an array on change. hasCheckbox adds tick boxes in the menu so users see what's already selected. multipleSelectMaxRows caps the chip stack before scrolling begins.

Source: props.md — isMulti, hasCheckbox, multipleSelectMaxRows; examples.md line 42.

❌ Don't — Stack several single Selects to fake multi-select

Stacked single Selects break the mental model, force redundant clicks, leave no "remove" affordance per choice, and produce no canonical array shape for the form. If the user picks several items for one slot, use isMulti.

Pair 3 — Use isAsync + loadOptions for remote or very large data
✅ Do — Stream options from the server when the full list is huge or external

When the option set lives in a backend (user directory, knowledge-base index, country lookup), isAsync swaps the static array for a loadOptions(inputValue, callback) function. The Select shows loadingMessage and sets aria-busy="true" while results are in flight.

Source: props.md — isAsync, loadOptions, loadingMessage; accessibility.md line 42. ⚠️ isAsync type conflict open — use boolean shorthand only until GAP-002 resolves.

❌ Don't — Pre-load 10 000 rows into options

Pre-loading a huge static list ships unnecessary bytes, makes the menu jank on open, and makes filtering feel slow. If the full dataset must ship client-side (offline mode), use VirtualMenuList via components{} instead — only visible rows render.

Source: examples.md lines 112–134 (VirtualMenuList).

Pair 4 — Always pair Select-in-Modal with menuPortalTarget + menuPlacement="auto"
✅ Do — Portal the menu out of the modal stacking context

Without a portal the popover is clipped by the modal's overflow rules or pushed behind its z-index. Capture the modal root via a callback ref and feed it to menuPortalTarget. Use menuPlacement="auto" so the menu opens upward when the trigger is near the viewport bottom.

Source: examples.md lines 74–110 (SelectModal); audit GAP-005 / GAP-006 (both props pending in props.md). Validate Modal portalRef against current @8x8/oxygen-modal (WARN-001).

❌ Don't — Drop a vanilla Select into a Modal

The menu will clip, overflow, or hide behind the modal mask. Users see a Select that "doesn't open." This is the single most reported regression on Select-in-Modal surfaces.

Pair 5 — Give every Select a meaningful labelValue
✅ Do — Visible label via labelValue; use infoBoxText for extra context

labelValue renders the visible label that htmlFor / aria-labelledby ties to the control. Use infoBoxText for the "why" — a tooltip the user opts into. The info button is a separate focus stop and must carry an accessible name via infoBoxButtonLabel, otherwise it's announced as unlabelled.

Source: props.md lines 83–84, 92; accessibility.md lines 40, 64.

❌ Don't — Rely on placeholder to label the field

placeholder disappears the moment a value is chosen — there is then no visible label at all, and assistive tech announces the control without a name. Dropping labelValue and showing the label as plain prose elsewhere breaks the htmlFor association.

Pair 6 — Pair hasError with a visible message — not a red border alone
✅ Do — Render hasError={true} and a sibling error element

hasError flips the visual error state and sets aria-invalid="true", but the error message is not part of the Select — consuming surfaces render it themselves below the field. Wire it in with aria-describedby where possible to satisfy WCAG 3.3.1.

Source: props.md line 77; accessibility.md lines 39, 63, 79. Audit GAP-007 — no error example yet in examples.md.

❌ Don't — Set hasError and call it done

A red border alone fails colour-only communication checks and leaves screen-reader users with no idea what is wrong. Users who don't perceive the colour will repeatedly retry the same submission.

Pair 7 — Be deliberate about isClearable on required fields
✅ Do — Set isClearable={false} when the field is required

isClearable defaults to true, so by default the field renders a clear × that drops the value back to nothing. On a required field that's a footgun — users can clear, blur, and submit an empty value. Either set isClearable={false} or pair clearing with explicit validation.

Source: props.md lines 86, 91 — isClearable default is true; isRequired sets aria-required.

❌ Don't — Leave isClearable at default on a required Select with no clear-handler

The default-true clear button silently invalidates the value with no user-visible feedback unless the form surrounds it with validation. The * required indicator advertises that a value is mandatory, but the affordance contradicts it.

Pair 8 — Keep placeholder short and distinct from the label
✅ Do — Use placeholder as a one-line hint about what to pick

The placeholder disappears the moment the user selects a value, so any information it carries is lost. Keep it to a short verb-phrase — "Select a region", "Choose a billing cadence" — and rely on labelValue for the persistent label and infoBoxText for context.

Source: props.md lines 92, 100; docs-page Anatomy (Label and Select field are separate parts).

❌ Don't — Stuff explanation into placeholder or duplicate the label

A long placeholder overflows the field, wraps awkwardly, and vanishes on select. Repeating the label inside the placeholder doubles the visual weight and adds nothing for screen-reader users — the labelValue label is already announced.
