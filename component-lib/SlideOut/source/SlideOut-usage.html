SlideOut — Usage Guidelines

No ↳ SlideOut examples Figma page exists (no Figma file exists at all for this component — see SlideOut-audit.md GAP-001 / GAP-002), so figma-extract-usage cannot run. The Do/Don't pairs below are inferred from props.md, examples.md, and accessibility.md via usage-advisor Open mode. Treat as provisional until a designer reviews them. SlideOut is a controlled side panel that animates in over the trailing edge of the page. Unlike a Modal, it does not block the rest of the UI; unlike a navigation Drawer, it is intended to be opened in response to a specific user intent and then dismissed. The component is deliberately thin — it renders a resizable container and a visibility flag, and expects the consuming application to wire up dismissal, focus management, and accessible labelling itself.

✅ When to use

  • Supplementary detail next to the main task — record details, conversation transcripts, configuration that complements the current view.
  • Inspectors / property panels the user opens, scans, and dismisses without leaving the page.
  • Side-by-side editing where the main canvas must remain visible and interactive.
  • Long content the user dismisses on their own schedule — help articles, audit logs, debug consoles.

❌ When not to use

  • Blocking decisions ("Are you sure you want to delete?") → use a Modal.
  • Persistent primary navigation → use a Drawer or sidebar.
  • Single-line hints → use a Tooltip.
  • Multi-step workflows that need their own URL or back-navigation → use a dedicated page or Modal route.
  • Mobile-first surfaces at very narrow viewports → use a Bottom Sheet or full-screen view.
SlideOut is fully controlled: isVisible is owned by the consuming application. The component does not ship a backdrop, focus trap, close button, header, footer, or Escape handler — see accessibility.md for the consumer's a11y checklist.

Do / Don't

Pair 1 — Pick SlideOut for supplementary side content; Modal or Drawer for everything else
✅ Do — Use SlideOut when the page behind it should stay visible and interactive

SlideOut renders no backdrop and does not trap focus, so the main view stays live. That is exactly right for companion content — an inspector, a help panel, a detail flyout — and exactly wrong for blocking decisions or permanent nav.

<Button onClick={() => setInspectorOpen(true)}>
  View details
</Button>
<SlideOut isVisible={inspectorOpen} isResizable>
  <RecordInspector
    recordId={selectedId}
    onClose={() => setInspectorOpen(false)} />
</SlideOut>

Grounded in: props.md:55-65 — no built-in scrim / focus-trap / close prop; accessibility.md:30-35 — role choice (see Pair 3).

❌ Don't — Use SlideOut for blocking confirmations or persistent navigation

A confirmation that needs an explicit choice belongs in a Modal — SlideOut has no backdrop and no built-in dismissal, so users can ignore it indefinitely. Persistent app navigation belongs in a Drawer.

{/* Wrong — destructive confirmation
   in a panel users can ignore */}
<SlideOut isVisible={confirmDeleteOpen}>
  <h2>Delete account?</h2>
  <Button onClick={deleteAccount}>Delete</Button>
</SlideOut>
Pair 2 — Control isVisible externally and reflect it on the trigger with aria-expanded
✅ Do — Drive open/closed from external state and mirror it on the trigger

isVisible is a controlled prop — the consuming application holds the boolean. The trigger that opens the panel should advertise the current state to assistive tech via aria-expanded.

const [isVisible, setIsVisible] = useState(false);

<Button
  aria-expanded={isVisible}
  aria-controls="details-panel"
  onClick={() => setIsVisible(true)}
>
  Show details
</Button>
<SlideOut id="details-panel"
          isVisible={isVisible}>
  …
</SlideOut>

Grounded in: props.md:58 — isVisible controlled; examples.md:62 — controlled-toggle pattern; accessibility.md:35 — aria-expanded on the trigger.

❌ Don't — Hard-code isVisible or open without a stateful trigger

There is no uncontrolled / defaultIsVisible mode. isVisible={true} leaves the user with no obvious dismissal path, and a trigger with no aria-expanded lies to screen readers.

{/* Wrong — always open, no way to close */}
<SlideOut isVisible>
  <RecordInspector recordId={selectedId} />
</SlideOut>
Pair 3 — Set role to match intent: dialog for interaction, complementary for information
✅ Do — Pick the role from the content's purpose; pair dialog with aria-labelledby

role="dialog" (with aria-labelledby) is right when the panel exists to gather input or confirm an action; role="complementary" is right when the panel is supplementary reference content the user can leave open while continuing to work.

{/* Interactive form-like content */}
<SlideOut isVisible={isVisible}>
  <section role="dialog"
           aria-labelledby="contact-form-title">
    <h2 id="contact-form-title">Add contact</h2>
    <ContactForm onSubmit={save} onCancel={close} />
  </section>
</SlideOut>

{/* Informational helper */}
<SlideOut isVisible={isVisible}>
  <aside role="complementary"
         aria-label="Field reference">
    <FieldReference fieldId={focusedFieldId} />
  </aside>
</SlideOut>

Grounded in: accessibility.md:30-35 — role choice; accessibility.md:49-50 — trigger label clarity.

❌ Don't — Use dialog for passive reference, or complementary for a form that demands a decision

Mis-roling the panel tells assistive tech the wrong story: dialog implies the user must interact before continuing, complementary implies the content is optional reading. Either mismatch confuses screen-reader users.

Pair 4 — Wire Escape, outside click, and focus return yourself
✅ Do — Implement the three dismissal paths in the consuming application

accessibility.md:43 is explicit: Escape "Should close the panel (must be implemented by the consuming application via isVisible state)." Same for focus return on close. Provide a keyboard handler, an outside-click handler, and a visible close affordance — and store the trigger so focus can return to it.

const triggerRef = useRef(null);
const panelRef = useRef(null);
const close = useCallback(() => {
  setIsVisible(false);
  triggerRef.current?.focus();
}, []);

useEffect(() => {
  if (!isVisible) return;
  const onKey = (e) =>
    e.key === 'Escape' && close();
  const onClick = (e) => {
    if (!panelRef.current?.contains(e.target))
      close();
  };
  document.addEventListener('keydown', onKey);
  document.addEventListener('mousedown', onClick);
  return () => {
    document.removeEventListener('keydown', onKey);
    document.removeEventListener('mousedown', onClick);
  };
}, [isVisible, close]);

Grounded in: accessibility.md:39-43 — keyboard contract; accessibility.md:45 — focus management is the consumer's responsibility; accessibility.md:58 — WCAG 2.1.2 (no keyboard trap).

❌ Don't — Assume the component handles dismissal, or omit a visible close affordance

Leaving Escape and outside-click unhandled traps keyboard users with no way to close the panel. Skipping a visible close button leaves pointer users guessing. Failing to return focus to the trigger drops the user at the top of the document on close.

Pair 5 — Toggle aria-hidden in lockstep with isVisible
✅ Do — Mirror aria-hidden on the panel when isVisible is false

The panel renders in the DOM regardless of visibility. Without aria-hidden, screen readers will announce the hidden panel's contents and may navigate into it via the rotor.

<SlideOut
  isVisible={isVisible}
  aria-hidden={!isVisible || undefined}
>
  <section role="dialog"
           aria-labelledby="filters-title">
    <h2 id="filters-title">Filters</h2>
    …
  </section>
</SlideOut>

Grounded in: accessibility.md:34 — aria-hidden="true" when isVisible=false; accessibility.md:49 — panel content rendered regardless of isVisible.

❌ Don't — Render the panel without an aria-hidden strategy

The panel's children stay reachable to assistive tech and to Tab order even when the panel is collapsed off-screen — both surfaces phantom content and creates a "where did my focus go?" moment for keyboard users.

Pair 6 — Turn on isResizable only when width is the user's lever
✅ Do — Enable resize for tabular, code, or canvas-shaped content; set minWidth / maxWidth

Resize earns its complexity when the user genuinely needs to control the panel's width — wide tables, code diffs, side-by-side previews — and is otherwise visual noise. Always bound the drag with minWidth / maxWidth so the panel cannot drag into a state that breaks the underlying layout.

<SlideOut
  isVisible={isVisible}
  isResizable
  defaultWidth={520}
  minWidth={360}
  maxWidth={960}
  onResize={(w) => persistPanelWidth(w)}
>
  <DiffView leftRef={baseRef}
            rightRef={headRef} />
</SlideOut>

Grounded in: props.md:59 — isResizable; props.md:61-63 — width props; props.md:73 — onResize (observed only — see GAP-005); examples.md:41-44 — canonical pattern; accessibility.md:51 — handle must be keyboard-operable with an accessible label.

❌ Don't — Ship resize on a short-form panel, or ship it unbounded

A resize handle on "Add comment" adds an affordance no user will reach for and a drag target keyboard users have to step past. Omitting minWidth / maxWidth lets users drag the panel into states that obscure the main view.

{/* Wrong — short form, unbounded resize */}
<SlideOut isVisible={isVisible} isResizable>
  <CommentForm onSubmit={save}
               onCancel={close} />
</SlideOut>
Pair 7 — Scope to a single concern; scroll internally; never nest
✅ Do — Keep the panel focused on one task and let internal scroll handle long content

A SlideOut works because it's narrow, dismissible, and parallel to the main task. Put one job in it (a record's details, one form, one document) and let the panel body scroll vertically when content overflows.

<SlideOut isVisible={isVisible}>
  <section role="dialog"
           aria-labelledby="record-title">
    <PanelHeader id="record-title"
                 title="Order #4821"
                 onClose={close} />
    <div style={{ overflowY: 'auto',
                  flex: 1 }}>
      <OrderSummary order={order} />
      <OrderTimeline events={order.events} />
    </div>
  </section>
</SlideOut>

Grounded in: "When not to use" above; accessibility.md:45 — focus management is the consumer's responsibility (nesting compounds it).

❌ Don't — Nest a multi-step workflow, scroll the page behind, or stack SlideOuts

A SlideOut hosting a four-step wizard breaks back-navigation and bookmarking — that flow wants a dedicated page or a Modal route. Opening a second SlideOut over the first compounds dismissal and focus-management work the component does not help with.

{/* Wrong — nested SlideOuts; dismissal
   and focus return become ambiguous */}
<SlideOut isVisible={outerOpen}>
  <RecordInspector />
  <SlideOut isVisible={innerOpen}>
    <EditField fieldId={focusedFieldId} />
  </SlideOut>
</SlideOut>
