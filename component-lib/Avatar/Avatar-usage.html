Avatar — Usage Guidelines

Avatar is a circular visual representation of a user, room, or other entity. It renders a photo, computed initials, or a fallback icon — in that priority order — and can carry a status badge, presence ring, and edit overlay depending on context and size.

✅ When to use

  • Identify a person, room, or entity at a glance in calls, contacts, conversation lists, or participant panels.
  • Pair with a userStatus badge to show presence (Available, Busy, Offline) for User-type entities.
  • Use AvatarStack when representing a group of related users with overlap and "+N" overflow.

❌ When not to use

  • Decorative illustration or product icon → use <img> or an Oxygen Icon
  • Group membership count with no specific members → use a numeric badge / pill
  • Brand or team logo in a non-contact context → use <img> with appropriate alt text

Do / Don't

Pair 1 — Interactive avatars need an accessible name
✅ Do — Wrap clickable avatars in a <button> with an aria-label

When an Avatar carries an onClick or onEdit handler it becomes an interactive control. The Avatar component provides no built-in ARIA role or label, so consumers must supply one. Use a native <button> (or role="button" + aria-label) so keyboard and screen reader users can discover and activate the control.

Grounded in: accessibility.md lines 62–69 (interactive ARIA guidance) · WCAG 4.1.2

❌ Don't — Pass onClick without any accessible label or wrapper role

An avatar with an onClick handler but no accessible name is invisible to screen readers and unreachable for keyboard-only users. Visual cues (cursor: pointer, hover highlight) are insufficient — the role and name must be in the DOM.

Pair 2 — Status information must reach screen readers
✅ Do — Include status in aria-label or add visually-hidden text

The status badge conveys its meaning through colour and an icon. Both are invisible to assistive technology by default. Include the status value in the avatar's accessible name, or use visually-hidden text alongside the component.

Grounded in: accessibility.md lines 100–113 (status/presence accessibility) · WCAG 1.3.1

❌ Don't — Show a status badge without communicating it to AT

Colour-only or icon-only status information fails WCAG 1.3.1. A screen reader user navigating a contact list will hear "Jane Smith" with no indication that she is busy or unavailable.

Pair 3 — Always pass name, even when a photo URL is provided
✅ Do — Provide name whenever you have it, regardless of src

name serves two purposes beyond showing initials: it supplies the text used in the initials fallback if the image fails to load, and it is the natural source for any accessible label on the avatar. Omitting it means a broken image silently degrades to a generic icon with no identification.

Grounded in: props.md line 70 (name computes initials) · props.md lines 114–121 (fallback chain: photo → initials → icon) · WCAG 1.1.1

❌ Don't — Omit name when a photo URL is available

If src is set but name is not, a broken or slow-loading image falls back directly to a generic icon — with no initials and no accessible text to identify the person.

Pair 4 — Reserve userStatus for User-type avatars
✅ Do — Apply userStatus only to avatars that represent a real user

Presence (Available, Busy, On Call, etc.) is meaningful for individual users. The Avatar component supports multiple entity types — User, Room, Auto Attendant, Call Queue, Ring Group, Channel, SMS Group — and only the User type has a Status variant in the Figma component set.

Grounded in: Avatar-figma.md lines 107–108 (Type variant axis) · lines 92–97 (non-User types have no Status axis)

❌ Don't — Add userStatus to Room, Call Queue, or other non-User avatars

Rooms and call queues don't have presence — showing a "Busy" badge on a conference room or an "Available" badge on an Auto Attendant is semantically misleading and has no counterpart in the Figma design system.

Pair 5 — hasStatusBorder requires a string userStatus
✅ Do — Pass a valid AvatarUserStatus string with hasStatusBorder

hasStatusBorder renders the status colour as a ring around the avatar. It is only activated when userStatus holds a valid AvatarUserStatus string (e.g. "Available", "Busy"). Passing a custom React element as userStatus disables the ring silently.

Grounded in: props.md line 75 (hasStatusBorder description) · line 130 (constraint)

❌ Don't — Combine hasStatusBorder with a custom React element userStatus

If you pass a ReactElement as userStatus, hasStatusBorder has no effect — no ring appears, and there is no error or warning. The constraint is completely silent.

Pair 6 — showEditOverlay is only available for sizes l–3xl
✅ Do — Use showEditOverlay on large (48 px) or bigger avatars

The edit overlay is designed for the profile-picture update pattern, where the avatar is large enough to accommodate a hover affordance. Sizes large through 3xl are supported. The xlarge (80 px) size is the most common context for edit overlays.

Grounded in: props.md line 79 · line 129 (constraint) · examples.md lines 103–117

❌ Don't — Enable showEditOverlay on xsmall, small, or medium avatars

The overlay is silently suppressed for sizes below large. Setting showEditOverlay={true} on a medium avatar produces no visible change and no console warning — making this a hard-to-spot misconfiguration.

Pair 7 — Use AvatarStack for groups; don't roll your own layout
✅ Do — Wrap related user avatars in AvatarStack

AvatarStack handles the overlap, z-index ordering, and "+N" overflow count that group representations require. It ensures visual consistency and saves you from reimplementing the same layout in CSS every time.

Grounded in: props.md line 60 (AvatarStack listed in package) · examples.md lines 167–178 (AvatarStack example)

❌ Don't — Manually lay out multiple avatars with negative margins

Hand-rolled overlap layouts diverge from the design system's intent for group representation and don't inherit future updates (overflow count, ring treatment, dark mode) that AvatarStack may introduce.
