Label is the textual identifier that names what a form control is asking for. It is the canonical pairing partner for Input, Checkbox, Select, and other Oxygen form components — rendered as a native HTML <label> element so it inherits the browser's built-in "click the label, focus the control" behaviour and screen-reader announcement. A well-formed Label is short, action-shaped, programmatically associated with its control via htmlFor, and consistent in placement and rhythm across the form surface.

⚠️ Open audit conflicts

Two conflicts in label-audit.md are unresolved at the time of writing. They do not block the editorial guidance below, but they constrain what you can confidently say about token authority and the info-icon API.

CONFLICT — Token name/size for label text. The Oxygen MCP returns label01 (Inter 12px / 16px line height, weight 400) at tokens.md §1. Figma resolves the same surface to typography/body01 (Inter 14px / 20px, weight 400) at Label-figma.md §3. One is authoritative for the React Label; the other is a parallel ramp used elsewhere. Pairs below avoid prescribing a specific type-ramp token and stay at the "use the canonical component, don't restyle" level.
CONFLICT — Figma information boolean ↔ Oxygen infoBoxText / infoBoxButtonLabel mapping. Label-figma.md shows a single boolean information that shows/hides an Icon Button slot. The React API exposes three related props — infoBoxText, infoBoxShowOn, infoBoxButtonLabel — and the exact mapping is unverified. Pairs about whether to use the info affordance are fine; pairs about which prop drives the visibility are deferred.

Anatomy

The Figma _base_form_label node composes the Label from up to four numbered parts plus an optional inline action exposed only via the React API:

  1. Label text — the visible string that names the associated control. Driven by the value prop. Rendered with text/textColor01 (#26252A Light / #FFFFFF Dark). MCP and Figma disagree on the type ramp — see Conflict above.
  2. Required asterisk * (optional) — shown when isRequired={true}. Rendered with error/error01 (#CB2233 Light / #F24D5F Dark). The asterisk is visual only — the programmatic required signal must come from aria-required="true" (or the native required attribute) on the associated form control.
  3. Info icon button (optional) — an Icon Button slot that opens an explanatory tooltip on hover or click. A separate keyboard focus stop with its own accessible name from infoBoxButtonLabel; trigger behaviour is controlled by infoBoxShowOn.
  4. Inline action link (optional, React only) — exposed via action, actionLabel, actionTarget, otherActionProps. Used for adjacent affordances like "Forgot password?" next to a password field. Not represented as a Figma variant on the current _base_form_label node.
  5. Overflow tooltip (showTooltipOnOverflow) — surfaces the full label text when truncated in a narrow container.
  6. Wrapped text (shouldWrapText) — allows the label string to break across multiple lines instead of truncating.
Layout. The Figma anatomy uses a horizontal H Stack with items-center, a 4 px hardcoded gap between the label text and the icon button, and the icon button wrapper itself has 2 px padding and a 6 px corner radius. The hardcoded values are flagged as missing spacing tokens.

Source: Label-figma.md §1, §3, §4 · props.md · accessibility.md.

Overview

A single Oxygen package, @8x8/oxygen-label, exports one component plus a typed export for actionTarget:

ComponentUse forKey props
Label The visible name of a form control — paired with Input, Checkbox, Select, etc. value, htmlFor, isRequired, isDisabled, shouldWrapText, showTooltipOnOverflow, action+actionLabel+actionTarget, infoBoxText+infoBoxShowOn+infoBoxButtonLabel
ActionTarget (secondary export) Type for the actionTarget prop — values like _blank, _self —

A Label renders an HTML <label> element. When htmlFor matches the id of a form control, the browser provides:

  • Click affordance. Clicking the label moves keyboard focus to the associated control.
  • Screen-reader announcement. Assistive technology announces the label text when the user reaches the associated control.
  • Programmatic association. The connection is exposed via the accessibility tree without extra ARIA — see WCAG 1.3.1.

Source: props.md · accessibility.md §1–§3 · every example in examples.md passes htmlFor.

Best Practices

When to use

  • To name a form control whose value the user is being asked to provide — input, textarea, select, checkbox, radio group, toggle.
  • When the control needs an inline secondary action that semantically belongs to "this field" (e.g. "Forgot password?" next to a password input).
  • When a field needs a brief explanatory tooltip that's longer than a placeholder allows but shorter than helper text deserves (e.g. "Where do I find this?").
  • For required-field signalling — pair isRequired on the Label with aria-required="true" (or the native required attribute) on the control.

When not to use (editorial — not on a published docs page)

SituationWhyUse instead
You need helper or error text below the field Helper text is the responsibility of the form control's own slot, not the Label The host component's helper/error slot
You need a placeholder Placeholders live inside the field's value affordance and disappear on focus — they are not labels placeholder on the control (sparingly) plus a real Label
You need a non-form heading <label> semantics are reserved for form controls; using it for arbitrary section titles confuses assistive technology A heading element (h2–h4) or Heading/Title typography component
You need a clickable affordance with no associated control <label> exists to forward focus — without an associated control its click behaviour is undefined A Button or Link

Do / Don't

These pairs are derived editorially from the extracted artifacts. No Figma ✅ Do / ❌ Don't card frames exist for Label at the time of writing, and the published docs page returned 404. Replace pairs verbatim if a Figma examples page is created with Do/Don't cards.

Pair 1 — Always associate the Label with its control via htmlFor

✅ Do — Pass htmlFor and match the id on the next control

The browser then routes label clicks to the input, and screen readers read "Email address, edit text" instead of just "edit text".

<Label value="Email address" htmlFor="email" />
<Input id="email" />

❌ Don't — Ship a Label with no htmlFor

The visual pairing is purely cosmetic. Sighted mouse users can aim at the field, but keyboard users get no focus hop and assistive technology has no programmatic relationship to read.

{/* Wrong — no association */}
<Label value="Email address" />
<Input id="email" />

Source: accessibility.md §3 (native <label> + htmlFor) · accessibility.md §10 (WCAG 1.3.1) · examples.md (every example shows the pairing).

Pair 2 — Use isRequired for the required signal, not text inside value

✅ Do — Set isRequired and let Oxygen render the indicator

The component renders the consistent * in error/error01 and design-system audits stay reliable. Pair aria-required="true" on the input so assistive technology hears the constraint, not just sees it.

<Label value="Password" htmlFor="password" isRequired />
<Input id="password" type="password"
  aria-required="true" />

❌ Don't — Bake the indicator into the label text

Inline * or "(required)" lose the token-driven colour, can't be localised independently of the label text, and turn into noise when an English screen reader reads "Password asterisk".

{/* Wrong — visual drift, no a11y signal */}
<Label value="Password *" htmlFor="password" />

{/* Wrong — verbose, inconsistent across the form */}
<Label value="Password (required)"
  htmlFor="password" />

Source: props.md (isRequired) · Label-figma.md §2 · accessibility.md §4.

Pair 3 — Pair isDisabled on the Label with isDisabled on the control

✅ Do — Disable the Label and the control together

Visual muting on the Label communicates "this field is off" to sighted users; isDisabled on the control supplies the disabled / aria-disabled="true" attribute that assistive technology actually uses to announce the state.

<Label value="Username" htmlFor="username"
  isDisabled />
<Input id="username" isDisabled />

❌ Don't — Disable only the Label and leave the control interactive

A muted Label paired with a live Input lies to sighted users about the field's availability and tells assistive technology nothing — the user can still tab into the input, type, and submit. The two states have to move in lockstep.

{/* Wrong — visual lies; control still focusable */}
<Label value="Username" htmlFor="username"
  isDisabled />
<Input id="username" />

Source: accessibility.md §5 (Label isDisabled is visual only) · props.md · examples.md.

Pair 4 — Pick one truncation strategy: wrap or tooltip-on-overflow, not both

✅ Do — Choose shouldWrapText for stacked forms, or showTooltipOnOverflow for tight grids

Wrap when vertical room exists and breaking the row rhythm is acceptable. Tooltip-on-overflow when horizontal rhythm matters and vertical drift would break the grid.

<Label value="A longer label that needs to wrap"
  htmlFor="my-input" shouldWrapText />
<Input id="my-input" />

{/* OR — for tight grids */}
<Label value="A very long label that might overflow"
  htmlFor="my-input" showTooltipOnOverflow />
<Input id="my-input" />

❌ Don't — Enable both, or leave both off when overflow is likely

Wrap defeats the tooltip-on-overflow trigger, so the tooltip never fires. Leaving both off in a tight container lets the browser hard-clip the string — sighted users lose the latter part of the label and keyboard users get nothing to fall back to.

{/* Wrong — wrap cancels the tooltip */}
<Label value="A very long label…"
  shouldWrapText showTooltipOnOverflow
  htmlFor="my-input" />

{/* Wrong — silent clipping */}
<Label value="A very long label that might overflow"
  htmlFor="my-input" />

Source: props.md (shouldWrapText, showTooltipOnOverflow) · examples.md §6, §7.

Pair 5 — Reserve the inline action link for one affordance that belongs to "this field"

✅ Do — Use one short action link for a per-field secondary affordance

A single inline action sits to the right of the field name and points at a destination semantically tied to that field — "Forgot password?", "Generate", "Reset", "Learn how to find this".

<Label
  value="Password"
  htmlFor="password"
  actionLabel="Forgot password?"
  action={() => navigate('/reset-password')}
/>
<Input id="password" type="password" />

❌ Don't — Stack multiple actions in the Label, or use it as a primary submit affordance

Stacking multiple inline links clutters the row and ambiguates the accessible name of the action. Primary actions belong in a Button, not embedded in the Label.

{/* Wrong — toolbar inside a Label */}
<Label value="Password"
  actionLabel="Forgot password? · Reset · Generate"
  action={...} />

{/* Wrong — Label is not a Submit button */}
<Label value="Save changes"
  actionLabel="Save"
  action={submitForm} />

Source: props.md (action, actionLabel, actionTarget) · examples.md §4 ("Forgot password?" pattern).

Pair 6 — Use the info icon for short context, not for instructions or required help

✅ Do — Put short "what does this mean?" copy in the info tooltip — and always supply infoBoxButtonLabel

Reach for the info icon when the user needs orientation about what the field is — provenance, expected format, where to find the value. Keep it to one or two sentences. infoBoxButtonLabel is required for the button to have an accessible name.

<Label
  value="API Key"
  htmlFor="api-key"
  infoBoxText="Your API key can be found in
    your account settings."
  infoBoxButtonLabel="More information about API key"
/>
<Input id="api-key" />

❌ Don't — Use the info icon for instructions, format hints, or error guidance

Instructions need to stay visible while typing — that's helper text on the form control. Error guidance must persist after focus moves on — that's the control's error slot. The info tooltip is "orientation on demand", not "guidance you need while typing".

{/* Wrong — format rules belong in helper text */}
<Label value="Password"
  infoBoxText="Must be 8+ characters, include one
    uppercase letter, one digit, one symbol…"
  infoBoxButtonLabel="Password rules" />

{/* Wrong — errors disappear with the tooltip */}
<Label value="Email"
  infoBoxText="That email is already registered."
  infoBoxButtonLabel="Email error" />

Source: accessibility.md §6 (info button is a separate focus stop; infoBoxButtonLabel required) · Label-figma.md §1 (Icon Button slot) · props.md. Pair addresses when to use the affordance, not which prop drives it (CONFLICT pending).

Pair 7 — Keep label text short, sentence-case, and free of trailing punctuation

✅ Do — Use 1–3 word labels in sentence case, with no terminal punctuation

Sentence case scans faster than Title Case, matches OS-level form conventions, and avoids capitalising prepositions inconsistently. Skipping trailing colons or periods keeps the visual rhythm clean and prevents screen readers from announcing "Email address colon".

<Label value="Email address" htmlFor="email" />
<Label value="Password"      htmlFor="password" />
<Label value="Phone number"  htmlFor="phone" />

❌ Don't — Write sentence-shaped or punctuated labels

Long, sentence-shaped labels stretch form rows unpredictably, fight with showTooltipOnOverflow, and waste screen-reader time on filler words. Reserve verbs and full sentences for helper text on the control.

{/* Wrong — instruction-shaped, sounds like a question */}
<Label value="Please enter your email address below:"
  htmlFor="email" />

{/* Wrong — Title Case + colon noise */}
<Label value="Email Address:" htmlFor="email" />

{/* Wrong — labels name fields, they don't narrate the action */}
<Label value="Enter password" htmlFor="password" />

Source: accessibility.md §11 (WCAG 2.4.6) · examples.md (every example uses 1–3 word labels with no terminal punctuation) · Input-usage.md Pair 7 (consistency).

Pair 8 — Stack the Label above the form control with consistent vertical rhythm

✅ Do — Place the Label directly above the control, with a consistent gap across the form

A single repeating Label-then-control rhythm makes the form scannable, lets sighted users predict where the next field begins, and gives keyboard users a reliable mental model of which Label belongs to which control.

<div className="form-field">
  <Label value="Email address" htmlFor="email" />
  <Input id="email" />
</div>
<div className="form-field">
  <Label value="Password" htmlFor="password" isRequired />
  <Input id="password" type="password" />
</div>

❌ Don't — Free-float Labels, mix patterns, or skip the Label entirely on "obvious" fields

Inline-left labels are a different design pattern than Oxygen's stacked default — mixing them within one form breaks visual scanning. Replacing the Label with a placeholder erases the affordance when typing starts and fails WCAG 3.3.2.

{/* Wrong — inline-left clashes with stacked default */}
<div style={{ display: 'flex' }}>
  <Label value="Email" htmlFor="email"
    style={{ width: 120 }} />
  <Input id="email" />
</div>

{/* Wrong — placeholder masquerading as a label */}
<Input id="email" placeholder="Email address" />

{/* Wrong — one Label, two controls, ambiguous */}
<Label value="Phone" />
<Input id="phone-country" />
<Input id="phone-number" />

Source: Input-usage.md (canonical form-row pattern) · examples.md (every snippet stacks Label above Input) · Label-figma.md §4 (the Label's internal auto-layout is horizontal; vertical placement is the host form's responsibility).

Accessibility quick reference

Cross-references to accessibility.md:

  • Renders a native HTML <label> — implicit role="none", programmatic association via htmlFor ↔ id.
  • Clicking the Label moves focus to the associated control — native browser behaviour, not a JS handler.
  • isRequired paints the visual * only. Programmatic required state must come from aria-required="true" (or the native required attribute) on the associated control.
  • isDisabled mutes the Label visually but does not suppress assistive-tech announcement; pair with isDisabled on the control to convey disabled state through aria-disabled.
  • The info button (infoBoxText + infoBoxButtonLabel) is a separate focus stop. Always supply infoBoxButtonLabel — without it the trigger has no accessible name.
  • When showTooltipOnOverflow is on, the tooltip content must be reachable by keyboard and announced by assistive technology.
  • WCAG 2.1 AA criteria addressed by Label: 1.3.1, 1.4.3, 2.1.1, 2.4.6, 3.3.2, 4.1.2.

See accessibility.md for the full WCAG 2.1 AA checklist.

Related components

  • Input — the most common Label pairing. Form-row rhythm, helper text, and error slot live on Input, not on the Label. See Input-usage.md.
  • Checkbox — bundles its own internal label via the label prop; do not wrap a Checkbox with this Label component. See checkbox-usage.md Pair 4.
  • Select — accepts a Label alongside it via htmlFor like Input.
  • Button — for primary actions or any clickable affordance that does not belong to a single form field; do not use the Label's inline action slot as a Button substitute. See Pair 5.
