Alert Banner — Usage Guidelines

AlertBanner is a persistent, page-scoped warning strip. It uses a single fixed amber style to signal that a system-level condition requires the user's attention. The banner stays visible until the underlying condition is resolved — it is not dismissable.

✅ When to use

  • A system-level warning affects the user's ability to proceed (session expiry, connectivity loss, scheduled maintenance).
  • The condition is persistent — not a one-time confirmation or transient status update.
  • The warning applies to the whole page or region, not to a specific field or isolated element.

❌ When not to use

  • Transient confirmation ("File saved") → use Toast
  • The user should be able to dismiss → use Toast (dismissable)
  • Field-level validation → use Inline error
  • Blocking action required → use Modal / Dialog
  • Low-urgency informational update → use Status region / Toast
AlertBanner has no built-in dismiss control. If the use case requires dismissal, use Toast.

Do / Don't

Pair 1 — Urgency match
✅ Do — Reserve for time-sensitive, page-level warnings

AlertBanner maps to an immediate screen reader announcement (role="alert" behaviour). Use it when the condition affects what the user can do right now — a session about to expire or a service outage blocking a core workflow.

Grounded in: accessibility.md — explicit role="alert" vs role="status" decision table.

❌ Don't — Use for low-urgency informational updates

Announcing "Autosaved 2 minutes ago" or "New features available" as an alert will interrupt screen reader users at the wrong moment. Use a polite status region or Toast for information that does not require immediate action.

Pair 2 — Single severity
✅ Do — Use only for warning-level conditions

The amber background is intentional and fixed. It signals "warning" system-wide. Using it only for that purpose preserves its meaning across the product.

Grounded in: props.md — no type, variant, or severity prop; single fixed amber style.

❌ Don't — Override its colour for success, info, or error states

There is no severity or variant prop. Applying custom CSS to change the background breaks the design system contract and produces an untokenised one-off. Raise a request with the design system team instead.

Pair 3 — Action button labelling
✅ Do — Label with a specific, self-contained verb phrase

"Retry", "Reconnect", "Sign in again", "Contact support" — each label tells the user exactly what will happen when they activate it, even when read in isolation by a screen reader.

Grounded in: accessibility.md — "meaningful out of context"; examples.md models "Retry" and "Contact support".

❌ Don't — Use generic labels like "Click here", "OK", or "More"

Generic labels fail screen reader users who navigate by button, and waste the action button's value for sighted users — the button is the clearest signal of what to do next, and a vague label squanders that.

Pair 4 — Message copy
✅ Do — Write short, user-actionable copy

"Your session has expired. Sign in again to continue." One sentence, two things: what happened and what to do. Keep children to plain text or simple inline elements.

Grounded in: props.md — "typically a plain text string… avoid complex interactive content"; examples.md models this pattern throughout.

❌ Don't — Paste raw error messages, status codes, or stack traces

Technical error strings are rarely actionable for end users and create noise for screen readers. If diagnostic information needs to be surfaced, link to a details view via the action button rather than embedding it in the banner message.
