Tag — Usage Guidelines

See also: props.md · examples.md · tokens.md · accessibility.md · Tag-figma.md · Tag-audit.md

Editorial guidance — not Figma-sourced. The Figma examples page for Tag (52829:11107) contains anatomy and type documentation, not ✅ Do / ❌ Don't card frames, so figma-extract-usage cannot run (audit reference: Tag-audit.md GAP-011). The Do/Don't pairs below are inferred from props.md, examples.md, accessibility.md, tokens.md, Tag-figma.md, and the public usage page at oxygen.8x8.com/components/tags/usage. Treat as provisional until a designer reviews them. Replace with figma-extract-usage output once Do/Don't card frames exist in Figma.

Tag is a pill-shaped component used to "label, categorize, or organize items using keywords that describe them." It has two visual forms — Standard Tag (with an optional leading icon) and Avatar Tag (with a fixed avatar) — selected automatically by the presence of the avatar prop. Either form can be display-only or carry a remove (×) button, controlled by the action prop.

✅ When to use

  • Categorize content, display metadata, filter items, or indicate status — the page's primary purpose.
  • As selected pills inside a multi-select Select — Avatar Tags or Standard Tags representing each chosen value, each with its own × button.
  • As a static status indicator on a card or list row — e.g. a blue "External" tag with no remove button (display-only).
  • As a RAG (Red / Amber / Green) status group — only on Standard Tag using variant="red" / "yellow" / "green".

❌ When not to use

  • You need a clickable affordance → use a Button. The tag body has no role or focus.
  • You want to convey count or numeric magnitude → use a Badge.
  • The content is a long phrase or sentence → re-think the data shape; Tag uses single-line label01.
  • You want a RAG colour on Avatar Tag → Avatar Tag has no yellow or green variants. Use Standard Tag.
  • You want a persistent emphasised warning → use variant="yellow-emphasis", not hasError.
  • You need a hover style → Tag has no hover/pressed states modelled.
Per oxygen.8x8.com/components/tags/usage: "Use tags to label, categorize, or organize items using keywords that describe them." Placement: "Tags should be 4px apart ($spacing02)."

Do / Don't

Pair 1 — Keywords, not actions; only removable when the user can remove it
✅ Do — Keep the label short and decide deliberately whether the tag should be removable

A Tag is a single-line 12 px label01 pill — use it for keywords, metadata, status words, or entity names. Pass action only when the user is meant to remove the item. For purely informational status (the public-usage-page "External" tag on a meeting card), omit action entirely so no × button is rendered.

{/* Display-only status tag */}
<Tag variant="blue">
  <GroupIcon />
  External
</Tag>

{/* Removable filter chip */}
<Tag
  action={() => removeFilter('happy')}
  actionProps={{ 'aria-label': 'Remove Happy filter' }}
>
  Happy
</Tag>

Grounded in: props.md 79–87 (children, action, actionProps); Tag-figma.md 60–73 (anatomy — remove button optional); accessibility.md 34–44 (dual-mode); public-usage-page placement-01 image ("External" tag with no ×).

❌ Don't — Wrap long phrases in a Tag, or attach action to a tag the user can't remove

A long sentence truncates and blows out the row — the pill is shaped for one-or-two-word labels at 12 px. Attaching action to a display-only status tag puts an × next to information the user is not meant to delete — the affordance lies.

{/* Sentence in a Tag */}
<Tag>Customer has not responded in 48 hours</Tag>

{/* × on a static status — user can't remove "External" */}
<Tag variant="blue" action={() => {}}>External</Tag>
Pair 2 — Avatar Tag for people/groups; Standard Tag for keywords/metadata/status
✅ Do — Use Avatar Tag for entities and Standard Tag for everything else

Avatar Tag's canonical home is a multi-select Select where each pill represents a user or group — see the public-usage-page placement-02 image (Avatar Tags as selected pills with × buttons inside a To-field). Standard Tag covers everything else: filter chips, status badges, metadata labels.

{/* Avatar Tag — multi-select Select */}
<Tag
  avatar={{ name: 'Josephine Lu', src: '/avatars/jlu.jpg' }}
  action={() => removeUser('jlu')}
  actionProps={{ 'aria-label': 'Remove Josephine Lu' }}
>
  Josephine Lu
</Tag>

{/* Standard Tag — keyword / metadata */}
<Tag variant="grey">Engineering</Tag>

Grounded in: props.md 81, 107–113 (avatar switches form); examples.md 102–123, 192–219 (Avatar Tag in multi-select Select); Tag-figma.md 311–314 ("Tags inside Select"); public-usage-page placement-02 image.

❌ Don't — Use Avatar Tag for status, or stuff a Standard Tag with an arbitrary leading photo

Avatar Tag's leading slot is the Avatar sub-component (24×24, rounded-1000px) — it expects a person/group avatar, not an icon or product thumbnail. Using it for status colour confuses readers because the photo carries entity identity, not status meaning.

{/* Avatar Tag used for status */}
<Tag
  avatar={{ name: 'Alert', src: '/icons/alert.png' }}
  variant="red"
>
  Critical
</Tag>
Pair 3 — RAG colours on Standard Tag only (yellow = "Amber" in design)
✅ Do — Build RAG status groups with Standard Tag

The Figma examples page names the pattern: "RAG tags — Red · Amber (yellow) · Green." The prop value is "yellow" but designers and Figma's color axis label it "Amber" (audit WARN-003). When a spec says "Amber tag", the code is variant="yellow".

<div style={{ display: 'flex',
              gap: 'var(--spacing/spacing02)' }}>
  <Tag variant="red">Critical</Tag>
  <Tag variant="yellow">Warning</Tag>  {/* "Amber" */}
  <Tag variant="green">Healthy</Tag>
</div>

Grounded in: examples.md 88–98 (RAG pattern); props.md 92–100 (yellow/green Standard-only); Tag-figma.md 131, 309 (Avatar Tag omits yellow/green; examples page names RAG); audit WARN-003 (yellow = "Amber").

❌ Don't — Apply RAG colours to Avatar Tag — yellow / green are not available

Avatar Tag's color axis ships only five values: default, blue, grey, red, yellow-emphasis — no yellow or green (Tag-figma.md line 131; audit GAP-008). If status colour belongs on a person/group, render the entity in Avatar Tag and put the colour on an adjacent Standard Tag.

{/* Avatar Tag has no `green` variant */}
<Tag
  avatar={{ name: 'Josephine Lu', src: '/avatars/jlu.jpg' }}
  variant="green"
>
  Josephine Lu
</Tag>
Pair 4 — Label the × button, not the tag body
✅ Do — Pass aria-label via actionProps on every removable tag

The tag body has no role, no tabIndex, and no keyboard interaction — only the × button is focusable. Figma's focus? boolean confirms this: the 2 px border + 4 px inset shadow is rendered around the close button area only, never the pill body. The × is icon-only — screen readers announce it as just "button" unless an accessible label is provided.

<Tag
  avatar={{ name: 'Josephine Lu', src: '/avatars/jlu.jpg' }}
  action={() => removeUser('jlu')}
  actionProps={{
    'aria-label': 'Remove Josephine Lu'
  }}
>
  Josephine Lu
</Tag>

Grounded in: accessibility.md 56–66 (ARIA — aria-label via actionProps), 86–95 (keyboard — only × takes Tab), 162–166 (missing aria-label is the top pitfall); Tag-figma.md 70, 84–86, 217–222 (focus ring on × area only); public-usage-page states image.

❌ Don't — Put aria-label on the tag body or leave the × button unlabelled

Setting aria-label on the Tag element does nothing — it has no role or tab stop. Rendering bare action={fn} with no actionProps announces "button" with no context — the user can't tell what removing it does until they activate it.

{/* aria-label on the tag body — ignored */}
<Tag aria-label="Remove Josephine Lu" action={remove}>
  Josephine Lu
</Tag>

{/* Bare action — no accessible label */}
<Tag action={remove}>Josephine Lu</Tag>
Pair 5 — Wrap tag lists in a flex row with gap: var(--spacing/spacing02) (4 px)
✅ Do — Use a 4 px gap between adjacent tags

The public usage page is explicit: "Tags should be 4px apart ($spacing02)." spacing02 is the same token Tag uses internally between leading visual, label, and × — so the rhythm carries across the row. Use a flex container with gap; don't put margins on individual tags.

<div style={{
  display: 'flex',
  gap: 'var(--spacing/spacing02)',
  flexWrap: 'wrap'
}}>
  {items.map((item) => (
    <Tag
      key={item.id}
      action={() => remove(item.id)}
      actionProps={{ 'aria-label': `Remove ${item.label}` }}
    >
      {item.label}
    </Tag>
  ))}
</div>

Grounded in: examples.md 172–188 ("$spacing02 (4px) apart"); tokens.md 114–120 (spacing02 = 4px, both internal gap and tag-to-tag spacing); oxygen.8x8.com — Placement: "Tags should be 4px apart ($spacing02)."

❌ Don't — Set margins on individual tags, or mix gap values across a row

Per-tag margins fight flex-wrap and collapse unpredictably across rows. Using gap: 8px or gap: 12px breaks the visual rhythm with Tag's internal spacing — the row no longer reads as part of the same component family.

{/* Per-tag margin + mixed gap */}
<div style={{ display: 'flex', gap: '12px' }}>
  <Tag style={{ marginRight: 8 }}>One</Tag>
  <Tag>Two</Tag>
</div>
Pair 6 — hasError is "retry this entity" on Avatar Tag, not a generic warning colour
✅ Do — Combine hasError with an action={retry} on Avatar Tag

hasError adds a red error/error01 border (Figma type=withActionError) — it's the Avatar-Tag-only state for "this person/group failed to load/send/sync and the user can retry." Pair it with an action that retries (not removes) and announce the error in text, not colour alone.

<Tag
  avatar={{ name: 'Failed user', src: '/avatars/failed.jpg' }}
  action={() => retry('failed')}
  actionProps={{
    'aria-label': 'Retry sending to Failed user'
  }}
  hasError
>
  Failed user
</Tag>

Grounded in: examples.md 128–141 (Avatar Tag error state); props.md 84, 121–125 (hasError); Tag-figma.md 130–132, 213–214 (Avatar Tag adds withActionError; error/error01 border); accessibility.md 66, 103, 166 (error must be conveyed in text, not colour alone).

❌ Don't — Use hasError as a generic warning colour or apply it on Standard Tag

A persistent warning colour belongs on variant="yellow-emphasis" (solid yellow). hasError is the transient "this Avatar Tag has failed" state. Per Tag-figma.md, withActionError is only defined on Avatar Tag — Standard Tag's behaviour with hasError is implementation-dependent.

{/* hasError as a generic warning */}
<Tag variant="default" hasError>Heads up</Tag>

{/* Emphasised warning — use the dedicated variant */}
<Tag variant="yellow-emphasis">Heads up</Tag>
