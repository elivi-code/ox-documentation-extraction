A Calendar lets users select a single date or a range of dates — past, present, or future — with optional time selection. It is the visual surface behind every date input in 8x8 product surfaces: scheduling, reporting windows, filters, and call-history queries. Two display modes share a single component, both driven by the displayMode prop.

Overview

Variant API Use for Notes
Single date displayMode="date" (default) Picking one specific calendar day Selecting a date closes the menu and populates the input immediately.
Date range displayMode="dateRange" Picking a start + end day, optionally with time inputs and a predefined-ranges panel First click sets startDate, second sets endDate. Change is committed by clicking Apply — not by the second click alone.

Source: Published docs — Overview section · props.md · Calendar-figma.md.

Anatomy

Single date calendar

  1. Navigation — ← previous-month, month dropdown, year dropdown, → next-month. Arrow buttons take their accessible name from prevMonthLabel / nextMonthLabel props.
  2. Days of the week — weekday header row, format controlled by weekDayDisplayFormat (default 'EE').
  3. Date grid — 40 × 40 px day cells. Today is marked with a dot; the selected day is filled with a blue circle.
  4. Title — month + year header, format controlled by monthDisplayFormat (default 'MMMM yyyy').
  5. Calendar menu — the popover container (border, 6 px radius, drop shadow). Opens from a date input on the consuming surface.

Date range calendar (additional elements)

  1. Predefined ranges panel (optional, 148 px wide) — "Today / Last hour / Yesterday / Last 7 days / This month / Last month / Last 3 months / This year / Last year / Custom".
  2. Date and time range inputs (optional) — From / To date inputs and Start / End time inputs, controlled by the Figma time boolean.
  3. Action bar — Reset · Cancel · Apply. Apply commits the range.

Source: Calendar-figma.md — anatomy section · Published docs — anatomy lists.

Sizes & Responsive Layout

The Calendar has no size prop — it adapts via months and breakpoint-driven layout shifts:

Breakpoint Predefined ranges Layout
≥ 769 px Optional Dual-month grid (default months={2})
541–768 px Optional Dual-month, compact
320–540 px Optional Single-month, stacked — set months={1}

Source: Calendar-figma.md — breakpoints section.

Placement

  • Calendars are popovers — they open from a date input and overlay the surface.
  • Single-date: selecting a date closes the menu and populates the input immediately — no Apply step.
  • Date range: the popover stays open until the user clicks Apply (or Cancel / Reset). This is intentional — a range is a two-click decision and intermediate state should not commit.
  • Don't render a Calendar inline without a triggering input — the popover affordance signals "this is editable".

Source: Published docs — Behavior sections.

Best Practices

When to use

  • When asking the user for an exact or approximate date (single or range).
  • When the user is required to schedule tasks (e.g. report generation, follow-ups).
  • When the user needs to pick a reporting window — the predefined-ranges panel is the fastest path for "Last 7 days", "This month", "Last quarter".

When not to use

Situation Why Use instead
The date is not required by the task Adds friction without value — reduce content on the page. Omit the field entirely.
A single date is required The two-click + Apply pattern overshoots a one-decision task. displayMode="date" — see Pair 1.
A free-form text date is needed ("around Q3", "TBD") Calendar enforces a concrete Date; placeholder semantics are lost. A plain text input or fuzzy-date picker.
Selecting a time only (no date) Calendar always picks a day — time inputs are an addition, not a substitute. TimePicker (see DateTimeSelector siblings).

Source: Published docs — Best Practices sections · props.md.

Do / Don't

Pair 1 — Pick the variant by selection type

Grounded in: Published docs — "When not to use" (Date Range Calendar) · examples.md (single and range examples).

✅ Do — Use displayMode="date" for a single date

When the task is "pick one day", the single-date variant is faster (one click, auto-close, auto-populate) and lighter on screen real estate (312 px vs 788 px).

<Calendar date={date} onDateChange={setDate} displayMode="date" />
❌ Don't — Reach for the range picker for a one-date task

The range variant requires two clicks then Apply. For a single date that's three interactions where one would do, and the Apply step blocks on a control they don't need.

{/* Wrong — range picker for a single-date task */} <Calendar range={range} onRangeChange={setRange} displayMode="dateRange" />
Pair 2 — Always label the month navigation buttons

Grounded in: Published docs — Anatomy (Navigation) · props.md (prevMonthLabel / nextMonthLabel) · accessibility.md · WCAG 4.1.2.

✅ Do — Pass localized prevMonthLabel and nextMonthLabel

The ← / → arrows have no visible text. The accessible name comes entirely from these props. Without them, screen readers announce the buttons as unlabelled — failing WCAG 4.1.2.

<Calendar date={date} onDateChange={setDate} prevMonthLabel="Previous month" nextMonthLabel="Next month" />
❌ Don't — Ship the calendar with bare arrow icons

Without prevMonthLabel / nextMonthLabel, the navigation buttons have no accessible name — screen-reader users hear "button, button" with no context.

{/* Wrong — nav arrows have no accessible name */} <Calendar date={date} onDateChange={setDate} />
Pair 3 — Use disabledDates + disabledDateTooltip for selection limits

Grounded in: Published docs — States (Disabled) · examples.md (disabled dates example) · accessibility.md · Calendar-audit.md GAP-011.

✅ Do — Drive limits through the API with a localized tooltip

The canonical pattern for the 90-day report window. The API has no dedicated maxRangeDays prop — build the window with disabledDates and surface the reason via disabledDateTooltip so keyboard and screen-reader users hear the constraint.

import { addDays, isBefore } from 'date-fns'; const windowStart = addDays(new Date(), -90); <Calendar displayMode="dateRange" range={range} onRangeChange={setRange} disabledDates={buildDisabledDates(windowStart)} disabledDateTooltip={(day) => isBefore(day, windowStart) ? '90-day report limit — earlier data is not retained' : undefined } />
❌ Don't — Hide the limit by silently disabling cells

A disabled cell with no tooltip tells the user nothing. They click, nothing happens, and they don't know why. Screen-reader users hear only "dimmed, button" with no reason.

{/* Wrong — limit enforced but never explained */} <Calendar displayMode="dateRange" range={range} onRangeChange={setRange} disabledDates={buildDisabledDates(windowStart)} />
🔖 GAP-011 (CONFLICT, resolved 2026-05-12 — Option A). The 90-day mode is a first-class Figma property (90 days boolean) without a dedicated API equivalent. Option A was chosen: document the disabledDates + disabledDateTooltip pattern as canonical. If Calendar gains a maxRangeDays prop in a future release, prefer that and update this pair.
Pair 4 — Require an explicit Apply for a range change

Grounded in: Published docs — Date Range Calendar > Behavior · Calendar-figma.md (action bar).

✅ Do — Keep the Reset / Cancel / Apply action bar in range mode

A range is a two-click decision (start + end). The Apply step gives the user a moment to confirm, cancel, or adjust before the value commits. It also gives keyboard users a stable focus target after the second click.

<Calendar displayMode="dateRange" range={range} onRangeChange={setRange} {/* The action bar (Reset · Cancel · Apply) is owned by the consuming DateTimeRangeSelector — don't auto-commit on the second click */} />
❌ Don't — Commit the range on the second click

Closing the popover on the second click removes the Cancel / Reset escape hatch, denies keyboard users a confirm step, and makes accidental range selection permanent with no recovery path.

{/* Wrong — second click commits without confirmation */} <Calendar displayMode="dateRange" onRangeChange={(range) => { setRange(range); closePopover(); // ← removes Cancel / Reset }} />
Pair 5 — Let locale and date-fns format the dates

Grounded in: Published docs — Date Format subsection · props.md (locale, dateOptions) · examples.md (French locale, Monday-first examples).

✅ Do — Pass a date-fns locale and let the component format everything

Date format expectations vary by locale ("the US uses M/D/Y, most other locations use D/M/Y" — published docs). Pass locale, set weekStartsOn, and let date-fns handle weekday and month names.

import { fr } from 'date-fns/locale'; <Calendar date={date} onDateChange={setDate} locale={fr} dateOptions={{ weekStartsOn: 1 }} // Monday-first />
❌ Don't — Override formatting with a hand-rolled US format

Hand-formatting dates leads to drift: "5/12/2026" reads as 12 May to a US user and 5 December to a UK one. Passing a hard-coded format string bypasses the locale system entirely.

{/* Wrong — assumes M/D/Y for every locale */} <Calendar date={date} onDateChange={setDate} monthDisplayFormat="M/d/yyyy" />
Pair 6 — Surface predefined ranges for common reporting windows

Grounded in: Published docs — Date Range Anatomy (Predetermined range) · Calendar-figma.md (predefined ranges panel, 148 px).

✅ Do — Use the predefined-ranges panel for "Today / Last 7 days / This month / …"

Most reporting flows reach for the same handful of windows. The predefined-ranges panel makes those one click instead of two-clicks + Apply. Reserve the day grid for cases that don't fit a preset.

<DateTimeRangeSelector predefinedRanges={[ { label: 'Today', value: () => today() }, { label: 'Last 7 days', value: () => last7() }, { label: 'This month', value: () => thisMonth() }, ]} range={range} onRangeChange={setRange} />
❌ Don't — Force hand-picking a window the panel supports

If the surface is a report filter and 80 % of users want "Last 7 days", surfacing only the day grid burns four clicks per query and adds extra tab stops for keyboard users.

Pair 7 — Choose layout by breakpoint, don't hardcode dual-month

Grounded in: Published docs — Anatomy (responsive layout) · Calendar-figma.md (breakpoints) · examples.md (single month view).

✅ Do — Collapse to a single month below 540 px

Two 280-px month columns plus the predefined-ranges panel exceeds phone width. Below 540 px, switch to months={1} and stack (or hide) the predefined-ranges panel.

const isNarrow = useMediaQuery('(max-width: 540px)'); <Calendar displayMode="dateRange" range={range} onRangeChange={setRange} months={isNarrow ? 1 : 2} />
❌ Don't — Hardcode months={2} on every surface

Two months plus a ranges panel at 788 px total width overflows any viewport narrower than ~830 px. On a phone it becomes horizontally scrollable or broken.

{/* Wrong — overflows narrow viewports */} <Calendar displayMode="dateRange" range={range} onRangeChange={setRange} months={2} />

Accessibility quick reference

  • The day grid renders as role="grid" with role="gridcell" children.
  • Today is marked with aria-current="date"; the selected day with aria-selected="true".
  • Disabled days carry aria-disabled="true" and remain in the tab order so the disabledDateTooltip is announced.
  • Tab, arrow keys, Enter / Space, Page Up / Page Down, Home / End are all wired. Follows the ARIA Date Picker pattern.
  • Always supply prevMonthLabel / nextMonthLabel (Pair 2) and locale (Pair 5).
  • Wrap the consuming surface in an aria-live="polite" region so the selection is announced.
  • WCAG 1.4.3 (Contrast) and 2.4.7 (Focus Visible) are marked Verify — they depend on tokens that are still unresolved (GAP-002, GAP-007).
