TimeSelector — Usage Guidelines

A TimeSelector lets a user pick a time of day — either by opening a dropdown and selecting a value, or by typing the time directly into the input. Reach for it in forms, scheduling flows, and report filters where the user needs to enter a specific hour and minute (and optionally an AM/PM marker or timezone).

✅ When to use

  • (Verbatim) When asking the user for an exact or approximate time.
  • (Verbatim) When the user is required to schedule tasks (e.g. reports).
  • Granularity is hours + minutes (and optionally AM/PM or timezone).
  • A free-text input would be ambiguous — the structured input + dropdown removes the ambiguity.

❌ When not to use

  • (Verbatim) When time is not necessarily required — drop the field.
  • Needs a date (year/month/day) → use Calendar.
  • Needs date + time together → compose Calendar + TimeSelector, or use a date-time pattern.
  • Needs a duration (90 min, 2h 30m) → use TextField with explicit parsing.
  • Small fixed set (9 AM / 12 PM / 5 PM) → use Radio or Select.
Two time-format rules from the docs page are non-negotiable: "If using the 12-hour format it must be accompanied by an AM/PM to follow" and "Specific times should specify a timezone." See Pair 3.

Do / Don't

Pair 1 — Time of day only
✅ Do — Use TimeSelector for hour-and-minute entry

The TimeValue shape — { hours, minutes, seconds } — is purpose-built for a single wall-clock moment within a day. Reach for it for meeting start times, report-window cutoffs, scheduling triggers.

Grounded in: props.md — TimeValue shape; timeselector-figma.md — input is "Time select", no date axis.

❌ Don't — Use TimeSelector for dates, date+time, or durations

There's no calendar axis in TimeValue and the dropdown has no day-picker affordance. Forcing TimeSelector into duration or date roles produces ambiguous, hard-to-validate input. Use Calendar for dates and TextField for durations.

Pair 2 — Controlled + pre-populated
✅ Do — Wire value + onChange as a controlled pair, and seed the value when you know it

The docs page is explicit: "The time will default to a placeholder … if a time has been previously selected then that time will prepopulate the time input field." Pass an initial TimeValue whenever you have one — last choice, system "now", or a sensible business default — and keep state in sync via onChange.

Grounded in: oxygen.8x8.com Behaviour section (verbatim); props.md — value / onChange; examples.md — Basic usage.

❌ Don't — Render <TimeSelector /> with no value and no onChange

The component is designed as a controlled input. Rendering it uncontrolled either freezes the input or causes the parent to lose track of what the user entered.

Pair 3 — Format + timezone discipline
✅ Do — Pair 12-hour displays with AM/PM, and surface the timezone when the value is timezone-sensitive

Two verbatim rules from the docs page: "If using the 12-hour format it must be accompanied by an AM/PM to follow" and "Specific times should specify a timezone." Put the AM/PM marker in the placeholder so the user knows what to type, and label the field (or a sibling hint) with the timezone the system will interpret the value in.

Grounded in: oxygen.8x8.com Time format section (both rules verbatim); props.md — timeDisplayFormat, placeholder.

❌ Don't — Show an unmarked 12-hour value, or hide the timezone for a timezone-sensitive moment

Without AM/PM, 5:00 is ambiguous (morning or evening?). Without a timezone label, 09:00 is ambiguous across regions — a problem that compounds when the value is persisted and read back in a different session.

Pair 4 — Error state + accessible message
✅ Do — Set hasError together with an explicit error message and a11y wiring

When validation fails, set hasError={true} so the input picks up the --error/error01 2 px border and exposes the Error Area below the input. Pair it with an error message linked via aria-describedby and set aria-invalid="true" on the input.

Grounded in: accessibility.md — Error state expectations; tokens.md — --error/error01 border token; timeselector-figma.md — Error Area visual.

❌ Don't — Style the input red without an accessible error message

Setting hasError alone gives sighted users a red border with no reason and gives screen-reader users no signal at all that the field is invalid. The red border alone also leans entirely on colour to convey state (fails WCAG 1.4.1).

Pair 5 — Disabled semantics
✅ Do — Disable when the field is meaningfully locked but readable

isDisabled={true} mutes the input background, mutes the text, and removes the input from the focus order. Use it when the field has a real value the user can read but currently can't change — for example, a scheduled-time field on a completed report.

Grounded in: props.md — isDisabled; accessibility.md — Disabled state expectations; tokens.md — disabled tokens.

❌ Don't — Disable an empty field as a "fill this in later" placeholder

A disabled empty input gives no affordance, drops out of the focus order, and may not meet contrast in the disabled state (WARN-001 in the audit). If the field isn't ready to be edited yet, hide it; if it's optional, leave it enabled with a placeholder.

Pair 6 — Match size to context
✅ Do — Use "default" for forms; reserve "small" for dense surfaces

The TimeSelectorSize enum exposes two values: "small" (32 px height, $label01 typography) and "default" (40 px height, $body01 typography). "default" is the right pick for stand-alone forms; "small" is appropriate for inline filters or tightly packed table cells.

Grounded in: props.md — TimeSelectorSize enum; tokens.md — Size variants table.

❌ Don't — Author code against size="large" or assume Large is the default

Figma exposes a Large variant (48 px height, $body02) and uses it as the Figma default, but the code API has no matching enum value — size="large" is a TypeScript error, and <TimeSelector /> with no size prop falls back to Medium, not Large. Treat Large as design-only until GAP-001 / GAP-002 are resolved.
