Tooltip — Usage Guidelines

See also: props.md · examples.md · tokens.md · accessibility.md · Tooltip-figma.md · Tooltip-audit.md

Mixed source. The text sections below were extracted from the Figma ↳ Tooltip examples page. The Do/Don't pairs were authored from the canonical published Oxygen docs at oxygen.8x8.com/components/tooltip/usage because the Figma page contains no ✅ Do — / ❌ Don't — card frames (Tooltip-audit.md GAP-017). The two prior audit blockers — character limit and orientation default — were reconciled 2026-05-15 against the published docs.

The Tooltip component displays contextual information when users hover over or focus on a UI element. Tooltips are supplementary — they provide short, non-essential hints that help users understand the purpose or function of an element. Keep content to a single sentence or brief phrase; for richer or interactive content, reach for Popover; for critical, always-visible information, reach for helper text.

✅ When to use

  • Explaining the purpose of an icon-only button.
  • Providing additional context for truncated text.
  • Clarifying abbreviations or technical terms.
  • Describing disabled elements.
  • Adding context to elements that need more explanation (e.g. table headers).

❌ When not to use

  • Long paragraphs or detailed instructions.
  • Complex information that requires user interaction.
  • Critical information the user must read — use helper text.
  • Hosting links, buttons, or interactive elements — these are inaccessible to some users.
Defaults are tuned: opens after a delay=300 ms hover, dismisses on blur, orientation="top", showOn="hover". Override only when the viewport edge clips the default placement or a touch UI needs a click trigger. Max-width 320px; max length 140 characters in English.

Do / Don't

Pair 1 — Keep tooltip content to 140 characters of plain text
✅ Do — One short, descriptive sentence per tooltip

The published docs cap tooltip length at 140 characters in English and require plain text. The container's max-width is 320px, which is sized for short hints — not paragraphs.

<Tooltip title="Move this file to Trash.
  You can restore it within 30 days.">
  <IconButton
    icon={<TrashIcon />}
    aria-label="Delete file" />
</Tooltip>

Grounded in: oxygen.8x8.com/components/tooltip/usage — Behavior; Tooltip-figma.md — max-width 320px.

❌ Don't — Stretch a tooltip into multi-line or multi-paragraph help

A tooltip that needs scrolling, line breaks, or punctuation-heavy phrasing belongs in a Popover or in helper text under the input. Tooltips can be missed, so anything that has to be read should not live here.

{/* Wrong — paragraph-length tooltip */}
<Tooltip title="Deleting a file removes
  it from this workspace and from any
  shared links. Restoration is possible
  within 30 days from the Trash view…">
  <IconButton icon={<TrashIcon />} />
</Tooltip>
Pair 2 — Use Tooltip for supplementary context, never for critical information
✅ Do — Reach for Tooltip when the information is nice to have

Tooltips earn their place when the rest of the UI already conveys the action and the tooltip just clarifies why or what for — an icon button's intent, an abbreviation, the meaning of a column header. Per the docs, use them "when more information is useful in helping a user make decisions but is not critical."

<Tooltip title="Send invite to selected
  teammates">
  <IconButton
    icon={<MailIcon />}
    aria-label="Send invite" />
</Tooltip>

<Tooltip title="Average API response
  time in the last 24 hours">
  <span>Latency (avg 24h)</span>
</Tooltip>

Grounded in: oxygen.8x8.com/components/tooltip/usage — When to use / When not to use; accessibility.md — touch / no-hover users.

❌ Don't — Hide load-bearing information in a Tooltip

If the user must read it to complete the task, the tooltip is the wrong home. Required-field rules, validation errors, fees, dates, and legal disclosures belong in always-visible helper text or in a Popover the user must dismiss. Tooltips can't be reached on touch devices without a tap-and-hold and may not appear at all in some configurations.

{/* Wrong — required fee in a tooltip */}
<Tooltip title="A 2.9% processing fee
  applies to all card payments.">
  <Button>Pay with card</Button>
</Tooltip>
Pair 3 — Tooltip content is text, not interactive
✅ Do — Plain text only inside title

The published docs are explicit: "Do not put links or other interactive elements in tooltips." Tooltips dismiss on blur and may not be reachable by all input modes; any element inside them that needs to be clicked, tabbed to, or focused will be unreachable for some users.

<Tooltip title="This dashboard
  refreshes every 5 minutes.">
  <InfoIcon />
</Tooltip>

Grounded in: oxygen.8x8.com/components/tooltip/usage — Behavior, When not to use; accessibility.md — keyboard / SR reachability.

❌ Don't — Embed links, buttons, or rich interactive content

A "Learn more" link or a "Confirm" button inside title creates content that keyboard, screen-reader, and touch users cannot reliably reach. If the content needs interaction, use a Popover instead — Popovers are designed for interactive, persistent content.

{/* Wrong — link inside tooltip */}
<Tooltip title={
  <>Refreshes every 5 minutes.{' '}
    <a href="/docs">Learn more</a>
  </>
}>
  <InfoIcon />
</Tooltip>
Pair 4 — Pair Tooltip with aria-label on icon-only buttons
✅ Do — Give the trigger its own accessible name

Tooltip exposes its content via aria-describedby by default (disableDescribedBy=false), which describes the trigger but does not name it. An icon-only button still needs its own aria-label (or visible text) so assistive tech can announce what the control is, independently of the supplementary tooltip text.

<Tooltip title="Archive this conversation.
  You can restore it from Archived.">
  <IconButton
    icon={<ArchiveIcon />}
    aria-label="Archive conversation" />
</Tooltip>

Grounded in: props.md — disableDescribedBy default false; accessibility.md — icon-button labelling.

❌ Don't — Use Tooltip as a substitute for an accessible name

A bare icon button with only a tooltip leaves screen-reader users with nothing announced when the button first receives focus, and breaks for keyboard-only users who pass quickly through the focus order. Disabling describedBy compounds the problem by removing the only programmatic association.

{/* Wrong — no aria-label on the icon */}
<Tooltip title="Archive conversation">
  <IconButton icon={<ArchiveIcon />} />
</Tooltip>
Pair 5 — Trust the defaults; pick an alternative when Tooltip is the wrong shape
✅ Do — Ship with the defaults and reach for the right alternative when content doesn't fit

The defaults are tuned: orientation="top", delay=300, showOn="hover", dismiss on blur. Override only when the viewport edge clips the default placement or a touch UI needs a click trigger. When the content is critical, interactive, or rich, pick the right alternative.

Content shapeComponent
Short, supplementary hintTooltip
Required-field rules, always-visible guidanceHelper text
Rich text, links, or buttons the user can interact withPopover
Decisions or workflows that must block the pageModal

Grounded in: props.md — orientation default 'top', delay default 300, showOn default 'hover'; oxygen.8x8.com/components/tooltip/usage — Behavior, Alternatives.

❌ Don't — Tune the defaults to mask a content-shape problem

A 0 ms delay flickers; a multi-second delay frustrates. Forcing showOn="click" to keep a tooltip persistent suggests the content actually belongs in a Popover. Stretching orientation and offset to fit a paragraph means the content is too long.

{/* Wrong — 0 ms delay flickers */}
<Tooltip title="Send" delay={0}>
  <IconButton
    icon={<SendIcon />}
    aria-label="Send" />
</Tooltip>

{/* Wrong — click + interactive content;
   this is a Popover */}
<Tooltip showOn="click" title={
  <><p>Filter results.</p>
     <Button>Reset</Button></>
}>
  <Button>Filter</Button>
</Tooltip>

Alternatives

Helper text

Text placed beneath an input field that provides important or critical information needed to complete the action. Always visible. Use for required-field rules and other vital information; links are permitted.

Contextual information

Copy that lives in the surrounding UI (panel intros, section descriptions) and guides the user without requiring a hover or focus.
