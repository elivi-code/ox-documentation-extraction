Radio Button — Usage Guidelines

See also: props.md · examples.md · tokens.md · accessibility.md

The Do/Don't pairs below are derived from the published oxygen.8x8.com/components/radiobutton/usage page cross-validated against props.md, examples.md, tokens.md, and accessibility.md. Radio Buttons are input controls that let a user select exactly one option from a related group. Selecting a new option automatically deselects the previously selected one. Reach for them in form choices, plan pickers, side-panel filters, modal confirmations, and any setting where the choices are mutually exclusive and the full list is short enough to display at once.

✅ When to use

  • The user must pick exactly one option from a related set.
  • The full list of options is short enough to show at once (typically ≤ 5 options).
  • Seeing every option side-by-side helps the user choose — they're comparing alternatives, not searching.
  • Suitable surfaces: forms, modals, side panels, full pages (verbatim from the docs page).

❌ When not to use

  • User can select multiple options → use Checkbox (verbatim from the docs page)
  • Setting takes effect immediately on toggle → use ToggleButton
  • Many options + constrained space → use a single-select Select
  • Purely visual read-only indicator → use plain text, Tag, or an icon
Every RadioGroup should have a preselected default. The Oxygen docs page is explicit: "for default view for a radio button group is to have one of them preselected."

Do / Don't

Pair 1 — Single-select only
✅ Do — Use Radio for mutually-exclusive single-select

Each RadioGroup enforces single-select by tracking one value. Use it when picking option A must clear option B — billing cadence, plan tier, yes/no decisions, mutually-exclusive filters.

Grounded in: accessibility.md — role="radiogroup" + native radio inputs; props.md — RadioGroup value is singular.

❌ Don't — Use Radio when more than one option can be selected

If the user can choose several options simultaneously, the affordance lies: arrow keys silently switch the selection, and the radiogroup semantics misrepresent the constraint. The docs page is explicit: "If the user can select multiple options, use checkboxes instead."

Pair 2 — Preselect a default
✅ Do — Ship a RadioGroup with one option already selected

Pick the safest or most common option as the default and pass it as the group's initial value. The docs page treats this as the default view; examples.md reinforces it: "One option should always be preselected."

Grounded in: oxygen.8x8.com — "for default view … one of them preselected"; examples.md — basic vertical group call-out.

❌ Don't — Ship a RadioGroup with nothing selected

An unselected group makes the keyboard pattern awkward (Tab lands on the first option, but no value is committed), forces an extra step, and breaks the docs-page expectation. If no safe default exists, consider whether Select with a placeholder is a better fit.

Pair 3 — Group container + label
✅ Do — Wrap related Radios in a RadioGroup and give it an accessible name

RadioGroup supplies role="radiogroup" so arrow-key roving-tabindex works and the options are announced as a set. Pair it with a heading via aria-labelledby, or set aria-label on the container when no visible label is possible.

Grounded in: accessibility.md — ARIA roles table; props.md — RadioGroup manages selection across children.

❌ Don't — Free-float bare Radio components

Without RadioGroup, the radiogroup role is absent, arrow keys don't roving-tabindex, the shared name is missing, and the $spacing(2) rhythm between items breaks down. AT users get no signal that the items belong together.

Pair 4 — Short lists only
✅ Do — Use Radio when the full option set is visible at once

Radio shines when the user can scan all alternatives in one glance and the comparison is part of the decision — 2–5 plans, yes/no/maybe, billing cadence. The docs page's example surfaces (forms, modals, side panels) implicitly assume a short list.

Grounded in: oxygen.8x8.com Placement section ("use … determined by the space in the UI"); cross-validated against the Checkbox usage page "many options + space pressure" rule.

❌ Don't — Stack 10+ options as Radios

A long vertical Radio list overflows the surface, slows scanning, and makes arrow-key navigation tedious. For long, paginated, or searchable lists (e.g. country pickers), use a single-select Select — one row of UI plus type-to-search.

Pair 5 — Label vs infoBoxText
✅ Do — Keep label tight; put context in infoBoxText

The label is the option name; the info button — a separate focus stop with its own accessible name from infoBoxButtonLabel — hosts the "why". This keeps Radio rows scannable and lets users opt into more detail.

Grounded in: props.md — infoBoxText + infoBoxButtonLabel; accessibility.md — info button is a separate focus stop; examples.md — info-tooltip example.

❌ Don't — Pack the explanation into label

Long labels wrap (the overflow rule top-aligns the radio to the first line), defeat scanning, and force every user — sighted or not — to read the full sentence before they can compare options. Use infoBoxText for the long version.

Pair 6 — Horizontal layout discipline
✅ Do — Use isHorizontal only for binary / near-binary choices with short labels

Horizontal layout works best for two- or three-option choices whose labels fit comfortably side-by-side — Yes/No, On/Off, Annual/Monthly. The $spacing(2) (8 px) gap between items is tight; anything longer than ~12 characters per label will feel cramped.

Grounded in: tokens.md — $spacing(2) = 8 px group gap; examples.md — "Use horizontal layout when space is limited and options are short"; props.md — isHorizontal.

❌ Don't — Force a long-label list into a horizontal row

Horizontal layout breaks down once labels wrap or the row count exceeds three. The input circle top-aligns to multi-line labels — which only looks intentional when each label is a single line. Stick to the default vertical layout in those cases.
