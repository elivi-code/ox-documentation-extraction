Slider — Usage Guidelines

See also: props.md · examples.md · tokens.md · accessibility.md · slider-pui.md · slider-audit.md

Editorial guidance — not Figma-sourced. No ↳ Slider examples Figma page exists, so figma-extract-usage cannot run. The Do/Don't pairs below are inferred from props.md, examples.md, accessibility.md, tokens.md, slider-pui.md, and the public usage page at oxygen.8x8.com/components/slider/usage. Treat as provisional until a designer reviews them. Replace with figma-extract-usage output when a Figma examples page is created.

Slider lets users select a value on a horizontal track by dragging the thumb or clicking the track. Its canonical use is audio volume — the Figma reference pattern is literally named "Audio Volume" — and more broadly any continuous value where approximating ("loud enough", "around 60%") is acceptable. The component supports a single thumb, a two-thumb range (isMultiple), and a draggable filled track (isTrackDraggable).

✅ When to use

  • The user needs to choose a value from 0 to 100 percent.
  • Volume control — audio in / audio out, ringer, microphone gain.
  • Range selection with a meaningful min and max — price range, time window — using isMultiple.
  • The value is bounded and continuous, and the user is happy approximating to the nearest step.

❌ When not to use

  • A few discrete options → use Dropdown
  • The range is large (e.g. 0–10 000) → use a numeric input
  • High-precision values required → use a numeric input
  • The control is binary (on / off) → use Switch (inferred)
Per oxygen.8x8.com/components/slider/usage: "A slider is an element that allows users to select or change a value on the track. … Use a slider when the user needs to choose a value from 0 to 100 percent. Use the slider for volume control."

Do / Don't

Pair 1 — Continuous bounded values, not a few options
✅ Do — Use Slider for 0–100% selection or volume control

The Slider is the right control when the user is choosing a value on a continuous, bounded scale where "near enough" is acceptable. Volume is the canonical example — the user wants "louder" or "quieter", not exactly 73.

<Slider
  value={value}
  minValue={0}
  maxValue={100}
  step={1}
  ariaLabel="Ring volume"
  onChange={setValue}
/>

Grounded in: oxygen.8x8.com — "Use a slider when the user needs to choose a value from 0 to 100 percent. Use the slider for volume control."; examples.md Basic single-thumb pattern + Figma "Audio Volume" reference.

❌ Don't — Use Slider when the user is choosing between a handful of discrete options

If the choices are "Low / Medium / High" or "1× / 2× / 4× / 8×", a slider forces the user to drag-and-snap to a value that maps to a label. A Dropdown lets them pick the named option directly.

{/* ❌ Four-option playback speed — wrong control */}
<Slider value={2} minValue={1} maxValue={4}
        step={1} ariaLabel="Speed" />

Grounded in: oxygen.8x8.com — "If there are a few options to choose from. Consider using a dropdown instead."

Pair 2 — Approximate values vs. precision
✅ Do — Use Slider when "approximately right" is good enough

Sliders excel when the user can land on a value by feel — gain, brightness, opacity, volume. The fast drag motion lets them sweep across the range and stop where it sounds / looks / feels right.

<Slider
  value={value}
  minValue={0}
  maxValue={100}
  step={1}
  ariaLabel="Microphone gain"
  onChange={setValue}
/>

Grounded in: oxygen.8x8.com — Behavior description; examples.md volume example with step={1}.

❌ Don't — Use Slider for huge ranges or values that must be exact

If the user is entering a dollar amount, a date, or any value where being off by a unit matters, the slider makes them aim instead of type. Large ranges and high-precision values are explicit anti-patterns on the OX usage page.

{/* ❌ Exact-amount entry */}
<Slider value={value} minValue={0} maxValue={10000}
        step={0.01} ariaLabel="Amount" />

{/* ❌ Wide range — coarse aiming */}
<Slider value={value} minValue={0} maxValue={10000}
        ariaLabel="Bandwidth" />

Grounded in: oxygen.8x8.com — "If there is a large range to choose from." + "When high-precision values are required."

Pair 3 — Use isMultiple for ranges
✅ Do — Use range mode (isMultiple) when the user picks a band, not a point

When the user selects a window — a price range, a date range, a min/max volume — use the two-thumb mode with a [number, number] value. Both thumbs share one track, so the relationship between min and max is visually clear and a single drag can shift the band (isTrackDraggable).

const [range, setRange] =
  useState<[number, number]>([20, 80]);

<Slider
  isMultiple
  value={range}
  minValue={0}
  maxValue={100}
  step={5}
  onChange={setRange}
  ariaLabel="Volume range"
/>

Grounded in: props.md isMultiple + isTrackDraggable; examples.md Range two-thumb + Track-draggable examples.

❌ Don't — Render two single sliders side-by-side for a min/max pair

Two independent sliders let the user set min > max and force them to compare two separate tracks to understand the range. The two-thumb mode keeps the values on one axis and prevents min/max inversion automatically.

{/* ❌ Fragile, disconnected */}
<Slider value={min} onChange={setMin}
        ariaLabel="Minimum" />
<Slider value={max} onChange={setMax}
        ariaLabel="Maximum" />

Grounded in: props.md isMultiple ("Enables range (two-thumb) mode. When set, value must be [number, number].").

Pair 4 — Always label the slider
✅ Do — Pair the Slider with a visible label, or supply ariaLabel when no label is present

The thumb renders as role="slider" and screen readers will announce only aria-valuenow (a number) by default — "50", with no context. A visible label tells sighted users what they're adjusting; ariaLabel does the same for screen-reader users when no visible label is possible.

<div>
  <label htmlFor="ring-volume">Ring volume</label>
  <Slider
    value={value}
    onChange={setValue}
    ariaLabel="Ring volume"
  />
</div>

Grounded in: accessibility.md ARIA role section + Required ARIA attributes + WCAG 2.5.3 Label in Name; props.md ariaLabel ("Required when there is no visible label nearby").

❌ Don't — Ship a Slider with no visible label and no ariaLabel

Without either, screen readers announce a bare number when the thumb is focused — "Slider, 50, 0 to 100" with no clue what is being controlled. The slider is then unusable to assistive-tech users and fails WCAG 4.1.2 (Name, Role, Value).

{/* ❌ No label and no ariaLabel */}
<Slider value={value} onChange={setValue} />

Grounded in: accessibility.md WCAG 4.1.2 Name, Role, Value.

Pair 5 — Expand the hit area on touch
✅ Do — Use expandKnobAreaBy (and expandTrackAreaBy) to reach a 44×44 target on touch surfaces

The visible thumb is 16×16 px at rest. On touch, that's below the WCAG 2.5.5 Target Size minimum (44×44 CSS px). Both props enlarge the invisible hit region without changing the visual size — so the design stays compact but taps land.

<Slider
  value={value}
  onChange={setValue}
  expandTrackAreaBy="8px"
  expandKnobAreaBy="8px"
  ariaLabel="Ring volume"
/>

Grounded in: accessibility.md Touch / pointer + WCAG 2.5.5; props.md expandTrackAreaBy / expandKnobAreaBy; tokens.md 16px default thumb.

❌ Don't — Ship the 16px default thumb in a dense, touch-driven layout

A dense list of sliders (an audio mixer, a per-channel volume column) with the default 16px thumb is unhittable on phones — users tap, miss, drag the wrong row.

{/* ❌ Mixer column with default-size thumbs */}
{channels.map(c => (
  <Slider key={c.id} value={c.level}
          onChange={c.set} ariaLabel={c.name} />
))}

Grounded in: accessibility.md WCAG 2.5.5; tokens.md — 32px expanded thumb is mouse-only Hover/Active state.

Pair 6 — Show the value when accuracy matters
✅ Do — Render the current numeric value alongside the slider

Most real sliders sit between "rough" and "exact". Showing the value as text next to the track lets the user verify what they landed on without taking the focus away from the gesture. Subscribe to onChange and render the value in a sibling element.

<div style={{ display: 'flex',
              alignItems: 'center', gap: '12px' }}>
  <Slider
    value={value}
    minValue={0}
    maxValue={100}
    step={1}
    ariaLabel="Ring volume"
    onChange={setValue}
  />
  <span aria-hidden="true">{value}%</span>
</div>

Grounded in: oxygen.8x8.com high-precision anti-pattern; accessibility.md Screen reader guidance (aria-valuenow already announced — use aria-hidden to avoid double-reads); props.md onChange fires on every change.

❌ Don't — Hide the value entirely when the user needs to dial a specific number

A slider with no readout asks the user to estimate where on the track corresponds to 75. On dense or short tracks that's a guess. Either show the value or replace the slider with a numeric input.

{/* ❌ No readout, user can't verify 75 */}
<Slider value={value} onChange={setValue}
        ariaLabel="Volume" />

Grounded in: oxygen.8x8.com — high-precision anti-pattern; accessibility.md — sighted users deserve the same value-announce affordance screen readers already get.
