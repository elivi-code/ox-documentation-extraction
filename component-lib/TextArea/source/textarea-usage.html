Textarea is the canonical multi-line free-form input in Oxygen — paragraphs, comments, notes, descriptions, messages, code snippets. It renders a native HTML <textarea> with implicit textbox role and aria-multiline="true", vertically resizable by default, sized by rows. Use it whenever the expected response runs past a single sentence or the user may want to insert line breaks.

Anatomy

The Figma baseline variant (Mode=Light, Size=Large, State=Rest, Error=False, Filled=False) numbers these elements. Only the input box ships inside @8x8/oxygen-textarea; everything else is supplied by the consumer.

  1. Label (consumer-rendered) — visible text above the field, named via htmlFor ↔ id or aria-labelledby. Rendered with typography/body01 in text/textColor01. Figma slot: _base_form_label, controlled by Show Label boolean (default true).
  2. Required marker (consumer-rendered) — visible * beside the label for required fields. Rendered with typography/bodyBold01 in error/error01. The asterisk is decorative (aria-hidden="true"); the semantic signal is required + aria-required="true" on the textarea.
  3. Help icon (consumer-rendered, optional) — an IconButton in the label row that opens a tooltip. A separate keyboard focus stop with its own accessible name.
  4. Field — the native <textarea> rendered by the component. Container: bg=ui/ui05, px=16, py=12, border-radius=6 (all hardcoded). rows (default 3) controls visible height.
  5. Placeholder (optional, discouraged as a label) — typography/body02 in text/textColor02. Disappears the instant the user types — see Pair 2.
  6. Resize grip — the diagonal handle in the bottom-right corner. Visibility/direction controlled by the resize prop (default 'vertical'; see Pair 7).
  7. Hint text (consumer-rendered, optional) — short helper rendered below the field, linked via aria-describedby. typography/label01 in text/textColor02. Figma slot: _base_form_hint.
  8. Error area (consumer-rendered, optional) — triangle icon (16×16, error/error01, decorative) + error message text in typography/label01 / error/error01. Replaces the hint row when hasError={true}. Linked via aria-describedby with role="alert" so screen readers announce the message on appearance.
  9. Focus ring — 2px solid interactive/focus01 (#0056E0 Light) applied as an absolute overlay (border-radius 6px) on keyboard focus. Required by WCAG 2.4.7 — never suppress with outline: none.

Source: textarea-figma.md §3 (Anatomy) · accessibility.md · props.md. Hardcoded inner-field spacings (16px h-pad, 12px v-pad, 6px border-radius, 4px gap between rows) lack spacing-token bindings — flagged in textarea-audit.md.

Sizing

Unlike Input, the OX Textarea API has no size prop. Vertical size is controlled by rows (default 3); horizontal size by cols (or the container width). The Figma Size axis (Large/Medium/Small) does not map to an API prop — it appears to control internal padding/typography only, and the Oxygen team's stance on this is unresolved (CONFLICT in textarea-audit.md).

  • rows={2–3} — short notes, display-name overrides, one-sentence prompts.
  • rows={4–6} — paragraphs, addresses, short descriptions.
  • rows={8+} — long-form content: bug reports, feedback essays, code paste-ins.
  • Field padding: 16px horizontal, 12px vertical (hardcoded).
  • Border radius: 6px (hardcoded).

Source: props.md:83 (rows default) · textarea-figma.md · textarea-audit.md (CONFLICT around Size axis).

Best Practices

When to use

  • The user must enter multi-line free-form content — paragraphs, comments, notes, descriptions, messages, code snippets.
  • The expected value is unpredictable in length and may benefit from line breaks and soft-wrapping.
  • The field is part of a form pattern that mixes TextField, Select, Checkbox, etc., and one input requires more vertical space than a single line.

When not to use

SituationWhyUse instead
Single-line input — name, email, code, searchTextarea's rows={3} default signals "write a paragraph"; a one-line answer in a 3-row box wastes vertical space.TextField — fixed-height, single-line, ships with built-in prefix/suffix/icon affordances.
One option from a known listForces the user to type a value that should be picked.Select, Radio, or Checkbox — same boundary as TextField Pair 1.
Rich-text / formatted content (bold, links, mentions)Textarea is plain-text only — no formatting affordances.A dedicated rich-text editor — Oxygen does not ship one.
A date, time, or date rangeFree-text dates are error-prone; pickers enforce a valid format.DateTimeSelector / DateTimeRangeSelector.

Do / Don't

Eight pairs grounded in TextArea's extracted files. Four mirror TextField-usage.md verbatim where the decision is identical; three are TextArea-specific craft heuristics (no internal precedent) — flagged with the CRAFT badge.

Pair 1 — Use Textarea for multi-line free-form input; reach for TextField when one line is enough

✅ Do — Use Textarea when the user is writing more than a line

Comments, descriptions, notes, multi-line addresses, support messages, code paste-ins — anything where the user may need to insert line breaks or where the expected response runs past a single sentence belongs in Textarea. The default rows={3} is a strong hint that a paragraph is welcome.

import { Textarea } from '@8x8/oxygen-textarea';

<label htmlFor="feedback">
  What could we improve?
</label>
<Textarea
  id="feedback"
  rows={4}
  placeholder="Tell us what didn't work…"
  onChange={(e) =>
    setFeedback(e.target.value)}
/>

❌ Don't — Use Textarea for a value that fits on one line

A 3-row box signals "write me a paragraph." Asking for an email, a ZIP code, or a referral code inside a Textarea wastes vertical space and confuses users about how much text you expect.

{/* Wrong — single-line value
    in a 3-row Textarea */}
<Textarea
  id="email"
  placeholder="you@company.com"
/>

{/* Use TextField for single-line values */}
{/* <TextField label="Email" /> */}

Source: accessibility.md:32 (implicit aria-multiline="true") · props.md:83 (rows default 3) · TextField-usage.md Pair 1 — extended to single-line vs multi-line boundary.

Pair 2 — Always pair Textarea with a real <label>; never use placeholder as a label

✅ Do — Provide a programmatic label every time

Textarea does not render its own label — the consumer wires one up via htmlFor ↔ id, aria-label, or aria-labelledby. A persistent visible label is a WCAG 2.1 AA requirement (1.3.1, 3.3.2) and stays visible after the user starts typing. Use placeholder only for example formatting — the shape of the value the user is about to type — never to communicate what the field is for.

<label htmlFor="bio">Bio</label>
<Textarea
  id="bio"
  placeholder="A short intro you'd put on
   a conference badge…"
  rows={3}
  onChange={(e) => setBio(e.target.value)}
/>

❌ Don't — Rely on placeholder as the field's label

Placeholder disappears the moment the user starts typing, fails contrast in many themes, and is announced inconsistently by screen readers. A placeholder-only Textarea is invisible to anyone who tabs back to review the form.

{/* Wrong — placeholder-as-label,
    fails WCAG 1.3.1 and 3.3.2 */}
<Textarea placeholder="Bio" />

{/* Right — real label + placeholder
    as example */}
<label htmlFor="bio">Bio</label>
<Textarea id="bio"
  placeholder="A short intro…" />

Source: accessibility.md:42-55, 145 ("labelling is the consumer's responsibility"; WCAG 3.3.2 marked consumer responsibility) · TextField-usage.md Pair 2 — mirrored.

Pair 3 — Put hints and constraints in a description below the field, not in placeholder

✅ Do — Render hint text as a sibling node and link via aria-describedby

Textarea has no built-in description prop — the consumer renders a hint element below the input and links it via aria-describedby. Helper text rendered below the field stays visible as the user types, is announced when the field receives focus, and is the right place for "Max 500 characters", "Markdown supported", "Include steps to reproduce".

<label htmlFor="repro">
  Steps to reproduce
</label>
<Textarea
  id="repro"
  aria-describedby="repro-hint"
  rows={5}
  maxLength={2000}
/>
<span id="repro-hint">
  List the exact steps so we can
  recreate the issue.
</span>

❌ Don't — Pack constraints into the placeholder

Once the user types one character, the placeholder is gone — along with the constraint. If the rule matters enough to state, it matters enough to stay visible.

{/* Wrong — rule vanishes on first
    keystroke */}
<Textarea
  placeholder="Max 2000 characters.
    Markdown supported."
/>

Source: accessibility.md:60-73, 153 (aria-describedby + hint pattern) · textarea-figma.md "Show Hint" anatomy · TextField-usage.md Pair 3 — adapted to Textarea's wrapper pattern.

Pair 4 — Use isReadOnly for visible-but-non-editable values; use isDisabled only when the field is genuinely unavailable

✅ Do — Reach for isReadOnly when the user needs to see and copy the value but not change it

Read-only textareas stay in the tab order, are announced as "read only" by screen readers, and let users select and copy the value — ideal for generated reports, saved notes, system-supplied content. Disabled textareas are skipped by keyboard navigation entirely and silent to assistive tech.

<label htmlFor="report">
  Generated summary
</label>
<Textarea
  id="report"
  value={summary}
  isReadOnly
  rows={6}
/>

❌ Don't — Disable a Textarea just to display a value

A disabled textarea is silent to assistive tech and skipped by keyboard navigation. If a user can read the value with their eyes but a screen reader user can't reach it, the form is broken. Reserve isDisabled for temporarily unavailable fields (prerequisite not met) — and explain why in nearby helper text.

{/* Wrong — value won't reach
    screen reader users */}
<Textarea
  value={summary}
  isDisabled
/>

{/* Right — read-only keeps it
    in the tab order */}
<Textarea
  value={summary}
  isReadOnly
/>

Source: props.md:89-90 (isDisabled / isReadOnly) · accessibility.md:101-113 · TextField-usage.md Pair 4 — mirrored.

Pair 5 — Show errors with hasError + an accessible error message; never colour alone

✅ Do — Combine hasError with a descriptive error message linked via aria-describedby

hasError only applies the red 2px border (props.md:117 — "error message display is handled at the form-field wrapper level"). The error text must be rendered as a sibling, linked via aria-describedby, with role="alert" so screen readers announce it on appearance. WCAG 3.3.1 requires errors to be described in text — colour alone fails.

<label htmlFor="message">Message</label>
<Textarea
  id="message"
  hasError={hasError}
  aria-describedby={hasError
    ? 'message-error' : undefined}
  rows={4}
  value={value}
  onChange={(e) =>
    setValue(e.target.value)}
/>
{hasError && (
  <span id="message-error" role="alert">
    Message must be at least
    20 characters.
  </span>
)}

❌ Don't — Set hasError without rendering an accessible message

A red border with no text fails WCAG 1.4.1 (use of colour) and 3.3.1 (error identification). Users with low vision, colour-blindness, or in high-glare environments may not perceive the border, and even those who see it won't know what is wrong or how to fix it.

{/* Wrong — red border, no text */}
<Textarea id="message" hasError />

{/* Wrong — vague message */}
<Textarea hasError
  aria-describedby="err" />
<span id="err" role="alert">
  Invalid.
</span>

{/* Right — specific + actionable */}
<Textarea hasError
  aria-describedby="err" />
<span id="err" role="alert">
  Add at least one sentence describing
  the problem.
</span>

Source: props.md:91, 117 · accessibility.md:60-73, 117-121, 144 (WCAG 3.3.1 consumer responsibility) · textarea-figma.md error anatomy · TextField-usage.md Pair 5 — mirrored with wrapper framing.

Pair 6 — Size rows to the expected input length CRAFT

✅ Do — Pick rows based on how much text you actually expect

A "Title" Textarea at rows={3} and a "Detailed description" Textarea at rows={3} give the user the same visual signal for two very different asks. Use rows={2–3} for short notes, rows={4–6} for paragraphs, rows={8+} for long-form content like bug reports or feedback essays.

{/* Short note */}
<Textarea
  id="alias"
  rows={2}
  maxLength={80}
/>

{/* Long-form feedback */}
<Textarea
  id="incident"
  rows={8}
  maxLength={5000}
/>

❌ Don't — Leave every Textarea at the default rows={3} regardless of expected input

A 3-row box for a one-sentence note wastes space; a 3-row box for a multi-paragraph bug report forces the user to scroll their own input as they write. Either way, you've signalled the wrong amount of effort.

{/* Wrong — same height for inputs
    expecting very different amounts
    of text */}
<Textarea id="title" rows={3} />
<Textarea id="description" rows={3} />
<Textarea id="reproSteps" rows={3} />

No internal precedent for multi-line sizing; adapted from TextField-usage.md Pair 6 — "Input width should be proportional to the content."

Source: props.md:83 (rows default 3) · examples.md controlled patterns · craft heuristic.

Pair 7 — Lock resize to vertical or none inside structured forms; avoid horizontal and both CRAFT

✅ Do — Keep the default resize="vertical", or set resize="none" in tight layouts

Vertical resize lets the user grow the field for long inputs without breaking the form's column width — the default for good reason. In dense dashboards, fixed-width modals, or grid-aligned forms where any height change would push surrounding content unpredictably, resize="none" and a generous rows value is the right call.

{/* Default — vertical resize allowed */}
<Textarea
  id="bio"
  rows={4}
  maxLength={500}
/>

{/* Locked in a tight grid cell */}
<Textarea
  id="note"
  rows={3}
  resize="none"
  maxLength={280}
/>

❌ Don't — Allow horizontal or both-axis resize inside fixed-width form columns

resize="horizontal" and resize="both" let the user widen the field beyond its column, breaking the form's grid alignment and on responsive layouts pushing other inputs offscreen. There is no Oxygen layout where horizontal resize improves the user's outcome.

{/* Wrong — lets the user
    break the grid */}
<Textarea
  id="bio"
  resize="horizontal"
/>
<Textarea
  id="bio"
  resize="both"
/>

{/* Right — vertical (default)
    is safe inside columns */}
<Textarea id="bio" />

No internal precedent — TextField has no resize. Recommendation is craft judgment; flag for designer review.

Source: props.md:87, 100-110 (resize enum, default 'vertical') · textarea-figma.md resize-grip anatomy · craft heuristic.

Pair 8 — Show a live character counter when maxLength represents a hard limit users care about CRAFT

✅ Do — Surface a visible counter when the limit is a real constraint (SMS, bios, summary fields)

When maxLength is set to a value the user is likely to bump up against — and where running out of room matters (a 160-char SMS, a 280-char bio, a 500-char incident summary) — render a live N / maxLength counter as a sibling element. Update it on onChange and tie it to aria-describedby alongside the hint so it's announced contextually.

const MAX = 280;
const [bio, setBio] = useState('');

<label htmlFor="bio">Bio</label>
<Textarea
  id="bio"
  maxLength={MAX}
  value={bio}
  aria-describedby="bio-hint bio-count"
  onChange={(e) =>
    setBio(e.target.value)}
/>
<span id="bio-hint">
  A short intro for your profile.
</span>
<span id="bio-count" aria-live="polite">
  {bio.length} / {MAX}
</span>

❌ Don't — Set a long maxLength and rely on the browser's silent cut-off

The browser will simply refuse keystrokes past maxLength with no feedback. Users assume their input is being captured, then discover later that the last paragraph of their bug report vanished. Either show a counter, raise the limit, or split the field.

{/* Wrong — silent cap, no feedback */}
<Textarea
  id="bio"
  maxLength={280}
/>

{/* Wrong — long limit, no counter,
    users may push it */}
<Textarea
  id="incident"
  maxLength={5000}
  rows={8}
/>

Explicitly flagged in TextField-usage.md as GAP-USAGE-003 — no internal ruling exists yet. This pair would set the precedent.

Accessibility quick reference

  • Renders a native <textarea> — implicit ARIA role textbox with aria-multiline="true"; no role override needed.
  • Label is associated via htmlFor ↔ id, aria-label, or aria-labelledby — never rely on placeholder alone (Pair 2).
  • Tab / Shift+Tab move focus; Enter inserts a newline (does not submit the form, unlike <input>); Escape has no default behaviour.
  • hasError paints the red border only — pair with aria-describedby pointing to an error region with role="alert" (Pair 5).
  • Mark required textareas with required + aria-required="true"; the visual * in the label is decorative and should be aria-hidden="true".
  • isReadOnly keeps the value reachable and copyable; isDisabled removes the field from the focus order entirely (Pair 4).
  • Focus ring must clear 3:1 contrast (WCAG 1.4.11) — never suppress with outline: none (WCAG 2.4.7).
  • Character counters (Pair 8) should be tied via aria-describedby with aria-live="polite" so screen readers announce the remaining count without interrupting typing.

See accessibility.md for the full WCAG 2.1 AA checklist.

Related components

  • TextField — single-line counterpart. The primary alternative when the value fits on one line (Pair 1). Also the source of internal precedent for Pairs 2-5.
  • Input — the lower-level single-line input that TextField wraps.
  • Select / Radio / Checkbox — when the value set is finite.
  • Button — most often submits the form a Textarea belongs to. Note: Enter inside a Textarea inserts a newline, so submit affordances must be explicit (button click, Ctrl+Enter, etc.).
